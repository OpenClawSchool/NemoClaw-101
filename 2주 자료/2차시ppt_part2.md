### 슬라이드 4: L7 정책 — GET vs POST

**슬라이드 내용**
| 작업 | 메서드 | 위험도 | 정책 |
|---|---|---|---|
| 이슈 조회 | GET | 낮음 | ✅ 허용 |
| PR 생성 | POST | 높음 | ⚠️ 승인 |
| 메일 조회 | GET | 낮음 | ✅ 허용 |
| 메일 발송 | POST | 높음 | ⚠️ 승인 |
| 대량 삭제 | DELETE | 위험 | 🚫 금지 |

**시각적 제안**
- 레이아웃: 표 중심, 위험도 열에 색상 막대 (초록→노랑→빨강)
- 하단 핵심: "어느 사이트냐가 아니라, 어떤 행위냐가 기준"

### 슬라이드 5: 추론 라우팅 — 보통 vs NemoClaw

**슬라이드 내용**
| | 보통 방식 | NemoClaw |
|---|---|---|
| API 키 | 에이전트가 보유 | 호스트에 보관 |
| 모델 호출 | 직접 외부 호출 | inference.local → 라우팅 |
| 모델 변경 | 코드 수정 필요 | OpenShell에서 교체 |

- 이점: ① 비밀키 격리 ② 모델 교체 용이 ③ 팀 거버넌스

**시각적 제안**
- 레이아웃: 좌우 비교 (빨강 vs 초록)
- 하단에 3가지 이점을 아이콘 + 한 줄로 표현

### 슬라이드 6: 실전 시나리오 4가지

**슬라이드 내용**
| 시나리오 | ✅ 자동 허용 | ⚠️ 승인 필요 |
|---|---|---|
| 개인 비서 | 캘린더·뉴스 조회 | 메시지 발송 |
| 창업가 비서 | 고객 문의·CRM 조회 | 메일 발송·CRM 수정 |
| 전문가 | 문서 검색·요약 | 실제 발송·데이터 업로드 |
| 개발자 | 이슈·로그 조회 | PR·배포·인프라 수정 |

**시각적 제안**
- 레이아웃: 2×2 그리드, 시나리오별 아이콘
- 하단 원칙: "읽기는 넓게, 쓰기는 좁게"
