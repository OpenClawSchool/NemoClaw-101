# 3차시. 실전 워크플로우 설계

## 개인 비서, 창업가 비서, 전문가 에이전트를 실제 업무 흐름으로 설계하기

## 1. 왜 3차시는 워크플로우 설계가 핵심인가

2차시에서는 NemoClaw와 OpenShell의 구조, 샌드박스, 정책, 네트워크 통제를 배웠습니다.
하지만 구조를 이해했다고 해서 바로 실무에 적용되는 것은 아닙니다.

실제 현장에서는 이렇게 묻게 됩니다.

* 내 업무 중 무엇을 에이전트에게 맡길 수 있는가
* 어디까지 자동화할 수 있는가
* 어디서부터 사람 검토가 필요한가
* 어떤 업무부터 시작해야 실패하지 않는가

즉, 실행형 AI는 기능이 많다고 잘 쓰는 것이 아니라,
**업무 흐름 속에 올바르게 배치할 때 비로소 가치가 생깁니다.**

이번 차시의 목적은 NemoClaw를 “멋진 기술”로 이해하는 데서 끝내지 않고,
**내 일에 붙는 실전형 에이전트 설계도**로 바꾸는 것입니다.

---

## 2. 이 차시에서 배울 것

이번 차시에서는 다음 네 가지를 해냅니다.

첫째, 업무를 단순 작업이 아니라 흐름으로 보는 법을 익힙니다.
둘째, 개인 비서형, 창업가 운영 비서형, 전문가 에이전트형 시나리오를 설계합니다.
셋째, 각 시나리오에서 입력, 판단, 실행, 승인, 기록 단계를 구분합니다.
넷째, 자동화에 적합한 업무와 사람 검토가 필요한 업무를 구분합니다.

이번 차시가 끝나면 수강생은 최소한 다음 중 하나를 만들 수 있어야 합니다.

* 아침 브리핑 에이전트
* 고객 문의 정리 에이전트
* 개인 문서 기반 전문가 응답 보조 에이전트
* 운영 체크리스트 보조 에이전트

---

## 3. 워크플로우 설계란 무엇인가

많은 사람이 AI 활용을 생각할 때 “무슨 기능이 있지?”부터 봅니다.
하지만 실행형 에이전트에서는 기능보다 **흐름**이 중요합니다.

예를 들어 “메일을 정리한다”는 말은 기능이 아닙니다.
실제로는 다음 같은 흐름일 수 있습니다.

1. 받은 편지함을 읽는다
2. 중요도와 성격에 따라 분류한다
3. 답장이 필요한 메일만 추린다
4. 초안을 만든다
5. 사람이 검토한다
6. 승인되면 발송한다
7. 기록을 남긴다

이런 식으로 업무를 풀어야
어디까지 자동화할지, 어디서 승인받을지, 어떤 정책이 필요한지가 보입니다.

즉, 워크플로우 설계는
**업무를 입력 → 판단 → 실행 → 승인 → 기록의 흐름으로 재구성하는 작업**입니다.

---

## 4. 좋은 워크플로우의 기본 원칙

실전형 에이전트를 설계할 때는 다음 원칙을 기억하면 좋습니다.

### 4.1 작고 명확한 문제부터 시작한다

처음부터 “내 회사를 다 운영해줘” 같은 목표를 잡으면 실패합니다.
대신 “오늘 일정 요약”, “답장 필요한 메일 5개 분류”, “문서 3개 요약”처럼 좁고 반복적인 업무부터 시작해야 합니다.

### 4.2 읽기 중심 자동화부터 시작한다

처음에는 조회, 검색, 분류, 요약 중심으로 설계하는 것이 안전합니다.
실제 수정, 발송, 삭제, 업로드는 뒤로 미루는 편이 좋습니다.

### 4.3 승인 포인트를 반드시 둔다

실행형 AI는 쓰기 작업이 들어가는 순간 위험도가 올라갑니다.
따라서 발송, 삭제, 등록, 수정 같은 작업은 승인 단계를 두는 것이 기본입니다.

### 4.4 기록이 남는 구조로 설계한다

좋은 워크플로우는 “무엇을 했는지”가 남아야 합니다.
에이전트가 어떤 정보를 읽었는지, 어떤 판단을 했는지, 어떤 제안을 했는지를 확인할 수 있어야 개선이 가능합니다.

### 4.5 사람의 역할을 없애는 것이 아니라 바꾼다

실행형 AI의 목적은 사람을 완전히 빼는 것이 아닙니다.
사람이 하던 반복 정리·분류·초안 작성을 대신하고, 사람은 검토·판단·최종 승인에 더 집중하게 만드는 것이 핵심입니다.

## 5. 워크플로우 설계 프레임

수업에서는 아래 프레임을 기준으로 모든 시나리오를 설계하면 좋습니다.

### 5.1 입력

에이전트가 무엇을 받는가

예:

* 이메일
* 캘린더 일정
* 문서 파일
* 고객 문의
* 로그 데이터
* 체크리스트 요청

### 5.2 판단

에이전트가 무엇을 기준으로 분류하거나 판단하는가

예:

* 중요도
* 긴급성
* 답장 필요 여부
* 카테고리
* 우선순위
* 템플릿 매칭 여부

### 5.3 실행

에이전트가 실제로 하는 일은 무엇인가

예:

* 요약
* 분류
* 태그 제안
* 초안 작성
* 일정 정리
* 보고서 생성

### 5.4 승인

사람이 어디서 개입하는가

예:

* 발송 전 승인
* 수정 반영 전 승인
* 외부 전송 전 승인
* 고객 응답 전 승인

### 5.5 기록

결과를 어디에 남기는가

예:

* 노트
* 로그
* 문서
* 작업 리스트
* CRM
* 일일 브리핑 파일

이 다섯 단계가 있어야 워크플로우가 운영 가능한 수준으로 올라갑니다.

## 6. 시나리오 1: 개인 생산성 비서

### 6.1 왜 유용한가

개인 생산성 비서는 실행형 AI가 가장 빨리 가치를 보여주는 영역입니다.
사람들은 하루에도 여러 번 메일, 일정, 할 일, 문서를 오가는데, 대부분이 “판단보다 정리 비용”이 큽니다.

채팅 중심으로 처리하면 좋은 이유는 명확합니다.

* 앱을 여러 개 열지 않아도 됨
* 아침에 한 번에 상황을 브리핑받을 수 있음
* 필요한 것만 추려서 볼 수 있음
* 초안까지 받아보고 마지막 판단만 하면 됨

### 6.2 대표 업무

* 오늘 일정 요약
* 중요한 메일 선별
* 답장 필요한 항목 추리기
* 할 일 재정리
* 문서 파일 찾기
* 저녁 업무 회고 정리

### 6.3 예시 워크플로우: 아침 브리핑 에이전트

입력:

* 오늘 캘린더
* 읽지 않은 메일
* 어제 미완료 작업

판단:

* 시간순 일정 정렬
* 긴급 메일 식별
* 오늘 꼭 해야 할 일 추리기

실행:

* 아침 브리핑 생성
* 일정 충돌 표시
* 중요한 메일 요약
* 오늘 우선순위 3개 제안

승인:

* 기본적으로 읽기 중심이라 승인 없이 브리핑 제공 가능

기록:

* 오늘의 브리핑 문서 저장
* 일일 노트에 핵심 항목 기록

### 6.4 예시 워크플로우: 답장 보조 에이전트

입력:

* 받은 편지함의 새 메일
* 사용자 선호 응답 스타일
* 과거 응답 템플릿

판단:

* 답장이 필요한지
* 바로 답할 수 있는지
* 사람 판단이 필요한지

실행:

* 메일 3줄 요약
* 응답 초안 생성
* 보류/즉답/중요 분류

승인:

* 실제 발송은 사람 승인 후

기록:

* 초안 저장
* 답장 대기 목록 정리

### 6.5 자동화 적합 업무와 사람 검토 업무

자동화 적합:

* 일정 요약
* 메일 요약
* 할 일 분류
* 문서 검색
* 회의자료 정리

사람 검토 필요:

* 실제 회신 발송
* 중요한 약속 확정
* 외부 공유
* 일정 변경 확정

## 7. 시나리오 2: 창업가 / 1인기업 운영 비서

### 7.1 왜 유용한가

1인기업이나 소규모 창업자는 대부분의 시간을 “생산”보다 “운영 정리”에 씁니다.
고객 문의, 미팅 준비, 콘텐츠 일정, 문서 업데이트, 리마인드, 자료 정리 같은 일들이 하루를 잘게 쪼갭니다.

이때 실행형 에이전트가 유용한 이유는
단순히 답해주는 것이 아니라, **운영 업무를 채팅 중심으로 압축해주기 때문**입니다.

### 7.2 대표 업무

* 고객 문의 분류
* 미팅 준비 자료 정리
* 제안서 초안 구성
* 콘텐츠 캘린더 정리
* CRM 입력 후보 정리
* 후속 연락 체크리스트 생성

### 7.3 예시 워크플로우: 고객 문의 정리 에이전트

입력:

* 이메일/채팅으로 들어온 문의
* 상품/서비스 FAQ
* 응답 템플릿
* 고객 유형 기준

판단:

* 문의 유형 분류
* 긴급성 판단
* 영업/지원/제휴 여부 구분
* 자동 응답 가능한지 판별

실행:

* 문의 요약
* 카테고리 태깅
* 응답 초안 작성
* 담당 처리 순서 제안

승인:

* 고객에게 나가는 실제 응답은 승인 후 발송

기록:

* 문의 로그 작성
* CRM 후보 정보 정리
* 후속 연락 필요 목록 생성

### 7.4 예시 워크플로우: 투자자 미팅 준비 에이전트

입력:

* 미팅 일정
* 상대방 정보
* 기존 사업 문서
* 최근 업데이트 노트

판단:

* 이번 미팅 목적 정리
* 상대방 관심 포인트 추정
* 필요한 자료 선별

실행:

* 미팅 브리핑 작성
* 예상 질문 목록 생성
* 핵심 숫자 정리
* 후속 액션 초안 생성

승인:

* 외부 공유 문서는 반드시 검토 후 사용

기록:

* 미팅 노트 초안
* 후속 액션 리스트
* 후속 메일 초안 저장

### 7.5 자동화 적합 업무와 사람 검토 업무

자동화 적합:

* 문의 정리
* 미팅 브리핑
* 자료 탐색
* 초안 작성
* 일정/후속관리 정리

사람 검토 필요:

* 가격 제안 확정
* 계약 관련 답변
* 외부 공지 발송
* 고객 민감 이슈 대응

## 8. 시나리오 3: 전문가 에이전트 / 지식 비즈니스 비서

### 8.1 왜 유용한가

전문가, 강사, 컨설턴트, 코치, 작가, 운영자는 이미 많은 지식과 자료를 갖고 있지만, 그것이 대화 중 즉시 활용되지 못하는 경우가 많습니다.

이때 실행형 에이전트는 단순 질의응답을 넘어서
**개인 문서, 노트, 강의자료, 상담 기록, 템플릿을 바탕으로 일관된 실무 보조**를 수행할 수 있습니다.

즉, 전문성을 단순 저장하는 것이 아니라
**반복 가능한 업무 흐름으로 꺼내 쓰게 해주는 것**이 핵심입니다.

### 8.2 대표 업무

* 강의자료 탐색 및 요약
* 상담 전 사전 브리핑
* 반복 질문 응답 초안 작성
* 고객 맞춤 자료 추천
* 기존 콘텐츠 재가공 초안 생성
* 지식 템플릿 정리

### 8.3 예시 워크플로우: 상담 전 브리핑 에이전트

입력:

* 고객 정보
* 이전 상담 기록
* 관련 문서/노트
* 오늘 상담 주제

판단:

* 핵심 이슈 정리
* 이전 상담 대비 변화 확인
* 오늘 중점 포인트 추리기

실행:

* 상담 전 브리핑 문서 생성
* 예상 질문 생성
* 추천 자료 후보 정리
* 상담 후 체크 항목 제안

승인:

* 외부 전달 문서는 검토 후 발송

기록:

* 상담 준비 노트 저장
* 후속 작업 목록 생성

### 8.4 예시 워크플로우: 반복 질문 응답 보조 에이전트

입력:

* 자주 받는 질문
* 기존 답변 템플릿
* 문서/노트/강의자료

판단:

* 기존 템플릿 재활용 가능 여부
* 맞춤형 수정 필요성
* 근거 자료 연결 가능 여부

실행:

* 질문 요약
* 응답 초안 생성
* 근거 자료 추천
* 추가 확인 질문 제안

승인:

* 고객 발송 전 승인
* 민감한 조언이나 계약성 발언은 수동 검토

기록:

* FAQ 축적
* 자주 쓰는 답변 템플릿 업데이트 후보 정리

### 8.5 자동화 적합 업무와 사람 검토 업무

자동화 적합:

* 자료 검색
* 상담 준비 브리핑
* 초안 작성
* FAQ 정리
* 콘텐츠 재정리

사람 검토 필요:

* 맞춤 조언의 최종 확정
* 계약/법률/재무 관련 민감 답변
* 외부 발송
* 고객별 특수 상황 판단

## 9. 시나리오 4: 개발자 / 운영 보조 에이전트

### 9.1 왜 유용한가

개발자와 운영자는 하루에도 여러 시스템을 오갑니다.
이슈, 로그, 테스트 결과, 문서, 배포 상태, 체크리스트를 분산된 도구에서 보게 됩니다.

실행형 에이전트가 유용한 이유는
이 산만한 상태를 채팅 중심으로 모아서 “무엇이 중요한지 먼저 보이게” 해주기 때문입니다.

### 9.2 대표 업무

* 에러 로그 요약
* 이슈 우선순위 정리
* 변경점 요약
* 배포 전 체크리스트 정리
* 장애 대응 브리핑
* 반복 작업 문서화

### 9.3 예시 워크플로우: 이슈 브리핑 에이전트

입력:

* 이슈 목록
* 최근 변경 기록
* 우선순위 기준
* 팀 운영 규칙

판단:

* 긴급도 분류
* 담당자 배정 후보
* 중복 이슈 여부
* 오늘 처리 우선순위 판단

실행:

* 이슈 요약
* 상위 우선순위 목록 제시
* 중복/유사 항목 묶기
* 코멘트 초안 제안

승인:

* 실제 수정, 머지, 배포는 승인 후

기록:

* 일일 개발 브리핑 기록
* 후속 액션 목록 저장

### 9.4 예시 워크플로우: 장애 대응 보조 에이전트

입력:

* 로그
* 에러 알림
* 운영 매뉴얼
* 최근 배포 내역

판단:

* 장애 범위 추정
* 최근 변경과의 연관성 추정
* 우선 점검 대상 제안

실행:

* 상황 요약
* 의심 원인 후보 정리
* 확인할 체크리스트 생성
* 커뮤니케이션 초안 작성

승인:

* 복구 조치, 설정 변경, 배포 롤백은 승인 후

기록:

* 사고 기록 초안
* 회고 항목 후보 저장

### 9.5 자동화 적합 업무와 사람 검토 업무

자동화 적합:

* 로그 요약
* 이슈 정리
* 상태 브리핑
* 체크리스트 생성
* 문서 검색

사람 검토 필요:

* 코드 변경
* 배포
* 설정 수정
* 데이터 삭제/복구

## 10. 어떤 업무가 자동화에 적합한가

실행형 AI에 잘 맞는 업무는 다음 특징을 가집니다.

* 반복적이다
* 입력 형식이 비교적 일정하다
* 결과 형식이 예측 가능하다
* 읽기 중심이다
* 사람이 최종 검토하기 쉽다

예를 들면:

* 메일 요약
* 일정 브리핑
* 문서 검색
* 초안 작성
* 분류와 태깅
* 체크리스트 생성
* 리포트 초안

이런 업무는 자동화 효과가 큽니다.

---

## 11. 어떤 업무는 인간 검토가 필요한가

반대로 아래 특징이 강한 업무는 사람 검토가 반드시 필요합니다.

* 외부 발송이 포함된다
* 실제 변경이 일어난다
* 금전, 계약, 법적 책임이 걸려 있다
* 고객 관계에 직접 영향을 준다
* 삭제나 수정의 되돌리기 비용이 크다
* 상황 맥락이 복잡하다

예를 들면:

* 이메일 실제 발송
* 계약 문구 확정
* CRM 정보 수정
* 결제/환불 승인
* 배포
* 고객 민감 이슈 대응
* 외부 공지 게시

좋은 자동화는 사람을 빼는 것이 아니라,
**사람이 더 중요한 판단에 집중하도록 앞단을 정리해주는 자동화**입니다.

## 13. 강사용 진행 팁

강의에서는 수강생이 너무 큰 그림만 보지 않도록 계속 좁혀줘야 합니다.

예를 들어 이렇게 유도하면 좋습니다.

* “당신이 매일 반복하는 일 중 가장 귀찮은 것 하나만 잡아보세요.”
* “그 업무는 읽기입니까, 쓰기입니까?”
* “에이전트가 어디까지 하면 도움이 되고, 어디서부터는 사람이 봐야 하나요?”
* “그 결과물을 매일 아침 받을 건가요, 요청할 때 받을 건가요?”
* “이걸 잘 만들면 하루에 몇 분이 줄어드나요?”

이 질문을 통해 기술 수업이 아니라 **업무 설계 수업**으로 전환할 수 있습니다.

## 14. 오늘의 핵심 정리

오늘 배운 핵심은 단순합니다.

실행형 에이전트의 가치는 기능이 아니라 **워크플로우 설계**에서 나온다.
좋은 시작점은 크고 복잡한 업무가 아니라, 작고 반복적인 읽기 중심 업무다.
모든 워크플로우는 입력, 판단, 실행, 승인, 기록의 다섯 단계로 보면 설계가 쉬워진다.
개인 비서, 창업가 운영 비서, 전문가 에이전트, 개발자 보조 에이전트는 모두 같은 구조로 설계할 수 있다.
자동화는 많이 하는 것이 아니라, **검토 가능하게 하는 것**이 중요하다.
