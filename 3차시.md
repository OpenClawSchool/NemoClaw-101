# 3차시. 사내 실전 워크플로우 설계 및 다중 에이전트 구축

## 주제: 미션 컨트롤(DB) 구축, 7대 핵심 두뇌 파일 세팅, 3계층 협업 프레임워크

## 학습 목표

- 전산 담당자의 반복 업무를 읽기/쓰기/외부 전송으로 분류하여 에이전트 자동화 범위를 맵핑한다.
- Docker 내 PostgreSQL을 활용해 실시간 기록을 위한 중앙 통제 센터(미션 컨트롤)를 구축한다.
- '빌더-오케스트레이터-실행자'로 이어지는 3계층 협업 프레임워크를 이해하고 다중 에이전트 환경을 기획한다.
- `soul.md`, `boundaries.md` 등 7개의 핵심 마크다운 파일을 통해 에이전트의 영구적 규칙과 경계를 설정한다.

---

## 1. 업무 자동화 범위 맵핑 (Automation Mapping)

단일 에이전트에게 모든 일을 맡기기 전, 업무의 성격에 따라 자동화의 수위를 결정해야 합니다. 이를 **R/W/S 분류 체계**라고 합니다.

| 분류 | 특징 | 업무 예시 | 자동화 정책 |
| :--- | :--- | :--- | :--- |
| **읽기 (Read)** | 정보를 조회하고 분석하는 작업 | 이슈 조회, 로그 확인, 인프라 상태 모니터링 | **자동 허용 (Default Allow)** |
| **쓰기 (Write)** | 시스템 설정을 변경하거나 파일을 생성/수정 | PR 생성, 티켓 상태 변경, 설정 파일 수정 | **관리자 승인 후 실행** |
| **외부 전송 (Send)** | 외부 시스템이나 고객에게 데이터를 전달 | 외부 메일 발송, 고객사 알림, API 데이터 전송 | **관리자 승인 후 실행** |

> **핵심 원칙:** "읽기는 자유롭게, 쓰기와 전송은 신중하게." 이 원칙이 지켜져야 안전한 에이전트 운영이 가능합니다.

---

## 2. 미션 컨트롤(Mission Control) 구축

다중 에이전트 환경에서는 각 에이전트가 수행한 작업과 상태를 공유할 중앙 저장소가 필요합니다. 이를 '미션 컨트롤'이라 부르며, PostgreSQL DB를 통해 구현합니다.

### 2.1 왜 데이터베이스(DB)가 필요한가?
- **영속성:** 에이전트가 재시작되어도 과거의 작업 내역과 상태를 기억합니다.
- **협업:** 여러 에이전트가 하나의 DB를 공유하며 서로의 작업 진행 상황을 확인합니다.
- **가시성:** 관리자가 DB를 통해 현재 어떤 에이전트가 어떤 작업을 수행 중인지 한눈에 파악합니다.

### 2.2 PostgreSQL DB 세팅 (Docker)
```bash
# Docker를 이용해 PostgreSQL 서버를 실행합니다.
docker run --name nemoclaw-db -e POSTGRES_PASSWORD=your_password -p 5432:5432 -d postgres
```
- **테이블 구성:** 총 17개의 표준 테이블(Agents, Jobs, Logs, Memory, Tasks 등)을 생성하여 시스템의 운영 데이터를 체계적으로 관리합니다.

## 3. 3계층 협업 프레임워크

복잡한 미션을 효율적으로 처리하기 위해 '지휘-전략-실행'으로 이어지는 계층형 구조를 적용합니다.

1. **1계층 (Builder):** **전산 담당자(사람).** 최종 의사결정권자로서 미션을 정의하고 정책(Policy)을 수립하며 중요한 작업을 승인합니다.
2. **2계층 (Orchestrator):** **메인 AI 에이전트.** 빌더의 추상적인 지시를 분석하여 세부 계획을 세우고, 적절한 서브 에이전트에게 일을 배분합니다.
3. **3계층 (Executor):** **전문 서브 에이전트.** 특정 도구(예: GitHub API, 로그 분석기)에 특화되어 지시받은 기술적 작업(코드 수정, 데이터 추출 등)을 수행합니다.

---

## 4. 7대 핵심 두뇌 파일 (Brain Files)

에이전트의 인격, 가치관, 그리고 절대로 넘지 말아야 할 선(Hard Boundaries)을 정의하는 7개의 마크다운 파일입니다.

1. **soul.md:** 에이전트의 존재 목적과 핵심 가치관 (예: "너는 보안 가디언이다.")
2. **identity.md:** 에이전트의 페르소나, 이름, 소속 정보
3. **guidelines.md:** 말투, 응답 형식, 보고 주기 등 구체적인 행동 지침
4. **boundaries.md:** **가장 중요.** 엄격한 권한 경계 및 금지 사항 정의
5. **context.md:** 현재 프로젝트의 배경, 인프라 구성 등 상황 정보
6. **memory.md:** 장기 기억 저장소 및 과거의 주요 결정사항 기록
7. **tools.md:** 에이전트가 사용할 수 있는 도구 목록과 API 사용법 명세

## 5. 실전 실습: 다중 에이전트 워크플로우 기획

**과제: "긴급 장애 대응(Incident Response) 자동화 팀" 기획하기**

1. **업무 분류:** 장애 발생 시 필요한 업무를 R/W/S로 분류해 보세요.
   - 예: 로그 확인(R), 원인 분석 보고서 작성(R), 서버 재시작(W), 장애 전파 메일(S)
2. **계층 설계:**
   - **오케스트레이터:** 전체 상황을 파악하고 분석가와 조치가를 지휘합니다.
   - **분석 에이전트(Executor):** 로그와 DB 상태를 조회합니다.
   - **조치 에이전트(Executor):** 빌더의 승인을 받아 실제 수정 작업을 수행합니다.
3. **규칙 설정:** 조치 에이전트를 위한 `boundaries.md` 핵심 내용을 한 문장으로 적어보세요.
   - 예: "모든 시스템 수정 및 재시작 명령어는 실행 전 반드시 빌더의 수동 승인을 받아야 한다."
