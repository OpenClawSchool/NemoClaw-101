# 4차시. 안전한 자동화 및 운영 체계

## 보안 원칙, 승인형 자동화, 운영 정책 수립으로 실행형 에이전트를 오래 안전하게 굴리는 법

## 1. 왜 4차시는 “자동화”보다 “운영 체계”가 핵심인가

1차시에서 NemoClaw의 개념을 이해했고,
2차시에서 OpenShell sandbox와 정책, 네트워크 통제를 배웠고,
3차시에서 실제 업무 워크플로우를 설계했습니다.

하지만 여기서 끝나면 아직 “실전 운영” 단계는 아닙니다.

실행형 AI는 한 번 잘 작동하는 것보다,
**계속 안전하게 작동하는 것**이 더 중요합니다.

왜냐하면 실행형 에이전트는 단순히 답만 하는 챗봇이 아니라,
사용자의 디지털 환경에 연결되어 실제 행동을 수행할 수 있기 때문입니다.
이 말은 곧, 잘못 설계하면 편리함이 아니라 사고로 이어질 수 있다는 뜻이기도 합니다.

예를 들어 이런 문제가 생길 수 있습니다.

* 승인 없이 외부로 메시지가 발송된다
* 잘못된 정책 허용으로 민감 정보가 외부 API로 나간다
* 읽기 자동화와 쓰기 자동화를 구분하지 않아 수정 사고가 난다
* 차단 로그를 보지 않고 허용만 늘려서 정책이 점점 느슨해진다
* 에이전트가 쓸모는 있지만 아무도 책임지지 않는 상태가 된다

현재 공식 문서 기준으로 NemoClaw는 OpenClaw를 **OpenShell 컨테이너 안에서 더 안전하게 상시 실행**하도록 돕는 오픈소스 레퍼런스 스택이며, OpenShell은 샌드박스 컨테이너, 자격증명 저장 게이트웨이, 추론 프록시, 정책 집행을 담당합니다. 즉, 구조 자체가 “기능 추가”보다 “통제 가능한 실행”을 염두에 두고 설계되어 있습니다. ([NVIDIA Docs][1])

그래서 4차시의 핵심은 이것입니다.

**자동화를 많이 만드는 법이 아니라, 검토 가능하고 책임 있는 자동화 체계를 만드는 법.**

---

## 2. 이 차시에서 배울 것

이번 차시에서는 다음 네 가지를 완성합니다.

첫째, 실행형 에이전트 운영의 핵심 보안 원칙을 이해합니다.
둘째, 자동 허용 / 승인형 / 금지 작업의 기준을 세웁니다.
셋째, 실제 운영에 쓸 수 있는 정책 문서와 승인 체계를 만듭니다.
넷째, 개인용에서 팀용/비즈니스용으로 확장 가능한 운영 틀을 설계합니다.

수업이 끝날 때 수강생은 최소한 아래 결과물을 가져가야 합니다.

* 내 에이전트 운영 원칙 1장
* 자동 허용 / 승인형 / 금지 작업 목록
* 승인형 자동화 흐름도 1개
* 주간 점검 체크리스트 1개

## 3. 안전한 자동화란 무엇인가

많은 사람들이 자동화를 “사람 개입 없이 돌아가는 것”으로 생각합니다.
하지만 실행형 AI에서는 그 정의가 위험합니다.

좋은 자동화는 무인 자동화가 아닙니다.
좋은 자동화는 **통제 가능한 자동화**입니다.

즉, 다음 조건이 있어야 합니다.

* 무엇을 자동으로 할지 명확하다
* 무엇은 승인받아야 하는지 분명하다
* 무엇은 절대 자동화하면 안 되는지 정의되어 있다
* 누가 검토하고 누가 책임지는지 정해져 있다
* 결과와 로그를 나중에 되짚어볼 수 있다

현재 공식 문서 기준으로 NemoClaw는 deny-by-default 네트워크 정책을 사용하며, 샌드박스는 명시적으로 허용된 엔드포인트에만 접근할 수 있습니다. 허용되지 않은 목적지는 OpenShell이 가로채고 운영자가 TUI에서 실시간 승인 또는 거부할 수 있습니다. 이런 구조는 곧 “자동화는 가능하지만, 기본값은 통제”라는 운영 철학을 보여줍니다. ([NVIDIA Docs][2])

즉, 실행형 AI의 핵심은
**자동으로 돌게 만드는 것**이 아니라
**안전하게 자동으로 돌 수 있는 범위를 정하는 것**입니다.

## 4. 운영자가 가져야 할 기본 보안 원칙

### 4.1 최소 권한 원칙

에이전트는 필요한 것만 접근해야 합니다.
“혹시 필요할지도 몰라서” 넓게 열어두는 순간 운영 리스크가 커집니다.

OpenShell 보안 가이드도 기본적으로 최소 허용에서 시작하고, deny 로그를 보며 필요한 범위만 추가하는 방식을 권장합니다. 또한 추론 라우팅, 파일시스템 접근, 네트워크 접근, 실행 사용자 권한을 가능한 좁게 유지하는 방향을 기본값으로 둡니다. ([NVIDIA Docs][3])

### 4.2 기본 차단 원칙

실행형 AI는 “일단 열어두고 필요하면 막는 것”보다
“일단 막아두고 정말 필요한 것만 여는 것”이 맞습니다.

NemoClaw의 네트워크 정책은 공식 문서 기준으로 deny-by-default이며, 승인되지 않은 외부 목적지는 자동 허용되지 않습니다. ([NVIDIA Docs][2])

### 4.3 읽기와 쓰기를 분리하는 원칙

조회, 요약, 분류 같은 읽기 중심 작업과
발송, 수정, 삭제, 등록 같은 쓰기 작업은 위험도가 다릅니다.

OpenShell 보안 및 정책 문서는 endpoint 단위에서 `protocol: rest`와 규칙, 또는 `read-only`/`read-write` 접근 프리셋을 통해 HTTP 행위를 나눠 제어할 수 있다고 설명합니다. 이 말은 곧, 단순히 도메인 허용이 아니라 **행위 수준까지 분리해서 운영하라**는 뜻입니다. ([NVIDIA Docs][4])

### 4.4 자격증명 분리 원칙

비밀키를 샌드박스 내부에 직접 노출하지 않는 것이 중요합니다.

공식 문서 기준으로 에이전트는 샌드박스 안에서 `inference.local`에 요청하고, 실제 공급자 자격증명과 업스트림 엔드포인트는 호스트가 소유합니다. 샌드박스에는 원본 API 키가 전달되지 않습니다. ([NVIDIA Docs][5])

### 4.5 로그와 검토 원칙

좋은 운영은 “문제 없길 바란다”가 아니라
**문제가 났을 때 왜 그런지 알 수 있는 상태**를 만드는 것입니다.

OpenShell은 정책 위반 시 deny 로그와 L7 deny 로그를 확인할 수 있게 하고, 이 로그를 바탕으로 정책을 조정하는 흐름을 제시합니다. ([NVIDIA Docs][4])

## 5. 자동 허용 / 승인형 / 금지 작업으로 나누기

실무에서 가장 유용한 운영 프레임은
업무를 세 종류로 나누는 것입니다.

### 5.1 자동 허용 작업

반복적이고 저위험이며 읽기 중심인 작업입니다.

예:

* 일정 조회
* 메일 요약
* 문서 검색
* 회의자료 정리
* 이슈 요약
* 로그 요약
* 체크리스트 생성

이런 작업은 자동화 가치가 높고, 사고 가능성은 비교적 낮습니다.

### 5.2 승인형 작업

실제 변경, 발송, 등록, 수정이 일어나는 작업입니다.

예:

* 이메일 발송
* 고객 응답 전송
* CRM 수정
* Git push
* PR 생성
* 외부 API POST
* 문서 외부 업로드
* 일정 변경 확정

이런 작업은 에이전트가 초안과 제안을 만드는 데까지는 잘할 수 있지만,
최종 실행은 사람 승인 후로 두는 편이 안전합니다.

### 5.3 금지 작업

되돌리기 어렵거나, 금전/법적/보안 리스크가 매우 큰 작업입니다.

예:

* 자동 결제 승인
* 대량 삭제
* 미검토 대외 발송
* 민감 데이터 외부 업로드 자동화
* 운영 정책 우회
* 권한 상승 시도
* 루트 수준 시스템 변경
* 무제한 외부 네트워크 허용

OpenShell 보안 문서는 루트 사용자 실행을 거부하고, 시스템 경로의 광범위한 쓰기 권한을 제한하며, 프라이빗 IP/루프백/링크로컬에 대한 우회적 접근도 기본적으로 차단하는 방향을 설명합니다. 이런 기본값은 “무엇을 금지해야 하는가”를 설계할 때 좋은 기준이 됩니다. ([NVIDIA Docs][4])

## 6. 승인형 자동화는 어떻게 설계하는가

승인형 자동화는 단순히 “버튼 하나 더 누르는 것”이 아닙니다.
좋은 승인형 자동화는 다음 흐름을 가집니다.

1. 에이전트가 필요한 정보를 읽는다
2. 판단/분류/추천을 한다
3. 초안 또는 실행 계획을 만든다
4. 사람에게 보여준다
5. 사람이 승인/수정/거절한다
6. 승인된 경우에만 실행한다
7. 결과를 기록으로 남긴다

이 구조가 중요한 이유는 분명합니다.

* 자동화 속도는 유지하면서
* 사고 가능성은 줄이고
* 사람은 마지막 판단에 집중할 수 있기 때문입니다

특히 NemoClaw/OpenShell 구조에서는 허용되지 않은 목적지가 자동으로 나가지 않고, 실시간 승인 흐름으로 이어질 수 있으므로, 승인형 자동화 철학과 기술 구조가 잘 맞습니다. ([NVIDIA Docs][2])

## 7. 승인형 자동화 실전 예시

### 7.1 이메일 응답 보조

* 필요한 후속 조치 제안

사람이 하는 일:

* 민감 표현 검토
* 가격/정책 최종 확인

입력:

* 실제 롤백/배포/설정변경 결정

입력:

* 고객 이력

* 브리핑 생성

사람이 하는 일:

* 맞춤 조언 최종 결정
* 민감 발언 검토
* 대외 전달 승인

## 8. 운영 정책 문서는 왜 필요한가

많은 사람이 AI를 도입할 때 프롬프트만 만들고 끝냅니다.
하지만 오래 쓰려면 결국 문서가 필요합니다.

운영 정책 문서는 다음을 정리합니다.

* 에이전트의 목적
* 사용 가능한 데이터
* 허용된 작업
* 승인 필요한 작업
* 금지 작업
* 로그 검토 주기
* 책임자
* 사고 시 대응 절차

이 문서가 있어야 다음 문제가 줄어듭니다.

* “이것도 해도 되나요?”가 매번 달라짐
* 사람마다 허용 기준이 다름
* 사고가 났을 때 누가 판단했는지 अस्पष्ट함
* 처음에는 안전했는데 시간이 갈수록 느슨해짐

현재 공식 문서 기준으로 NemoClaw는 온보딩, lifecycle management, OpenClaw operations를 OpenShell 컨테이너 안에서 관리합니다. 즉, 단발성 실행이 아니라 **계속 굴리는 운영 대상**으로 보는 것이 맞고, 그래서 정책 문서화가 중요합니다. ([NVIDIA Docs][1])

## 10. 점검 체크리스트 만들기

실행형 에이전트는 한 번 세팅하고 끝내면 안 됩니다.
정기 점검이 필요합니다.

### 10.1 일일 점검

* 이상한 발송 시도는 없었는가
* 승인 대기 항목이 너무 쌓이지 않았는가
* 자동 요약 품질이 떨어지지 않았는가
* 사용자 불만이나 오탐이 있었는가

### 10.2 주간 점검

* deny 로그에서 반복 차단되는 요청은 무엇인가
* 정말 필요한 엔드포인트만 허용되어 있는가
* 승인 기준이 너무 느슨해지지 않았는가
* 에이전트가 다루는 데이터 범위가 커지지 않았는가
* 초안 품질이 업무에 실질적으로 도움이 되는가

### 10.3 월간 점검

* 금지 작업 목록을 다시 봤는가
* 더 이상 쓰지 않는 허용 정책이 남아 있지 않은가
* 책임자와 승인자가 명확한가
* 개인용 운영에서 팀용 운영으로 확장할 준비가 되었는가

OpenShell 보안 문서는 승인된 엔드포인트가 같은 sandbox 인스턴스 내에서는 재시작 후에도 지속될 수 있고, sandbox를 재생성하면 초기화된다고 설명합니다. 이런 특성 때문에 정기적으로 “이미 넓어진 허용 범위”를 다시 검토하는 습관이 중요합니다. ([NVIDIA Docs][4])

## 11. 개인용에서 팀용/비즈니스용으로 확장하는 법

처음에는 개인 비서형으로 시작하는 것이 좋습니다.
이 단계에서는 본인이 직접 운영자이자 승인자입니다.

하지만 팀이나 비즈니스용으로 가면 달라집니다.

### 11.1 개인용 운영

* 사용자와 승인자가 같은 경우가 많음
* 빠른 실험이 가능함
* 범위를 좁게 유지하면 효과를 빨리 볼 수 있음

### 11.2 팀용 운영

* 누가 승인자인지 분리해야 함
* 공용 데이터 접근 범위를 명확히 해야 함
* 역할별 허용 작업이 달라질 수 있음
* 로그/감사/정책 문서화가 더 중요해짐

### 11.3 비즈니스용 운영

* 고객 데이터, 계약, 대외 커뮤니케이션이 들어오기 시작함
* 무조건 자동화보다 승인형 워크플로우가 기본이 되어야 함
* 운영 정책, 책임 체계, 기록 보존이 필수에 가까워짐

즉, 확장의 방향은 “더 많은 자동화”가 아니라
**더 정교한 승인 구조와 거버넌스**입니다.

## 12. 수강생 실습

### 실습 1. 자동 허용 / 승인형 / 금지 표 만들기

본인이 설계한 3차시 워크플로우를 기준으로 아래를 채웁니다.

* 자동 허용 작업 5개
* 승인형 작업 5개
* 금지 작업 5개

### 실습 2. 승인형 자동화 흐름도 만들기

아래 단계를 포함해 흐름도를 작성합니다.

* 입력
* 판단
* 초안 생성
* 사람 검토
* 승인/거절
* 실행
* 기록

### 실습 3. 운영 정책 1장 작성

다음 문장을 완성합니다.

* 내 에이전트는 무엇을 위해 존재하는가
* 무엇까지 자동으로 해도 되는가
* 무엇은 반드시 승인받아야 하는가
* 무엇은 절대 하면 안 되는가
* 누가 최종 책임자인가
* 로그는 얼마나 자주 볼 것인가

### 실습 4. 사고 시나리오 대응 훈련

아래 중 하나를 골라 대응 방안을 적습니다.

* 잘못된 메일 초안이 자동 발송 직전까지 감
* 허용되지 않은 외부 API 요청이 반복 발생
* 요약 결과가 중요한 맥락을 빠뜨림
* 외부 업로드 요청이 들어왔는데 정책상 불명확함

실습의 핵심은 “기능 설명”이 아니라
**문제가 생겼을 때도 운영할 수 있는가**를 훈련하는 데 있습니다.

## 13. 강사용 진행 포인트

강의에서는 아래 메시지를 반복해서 강조하면 좋습니다.

첫째, 자동화할수록 좋은 것이 아니다.
둘째, 읽기 자동화와 쓰기 자동화는 반드시 나눠야 한다.
셋째, 정책은 귀찮은 문서가 아니라 사고 방지 장치다.
넷째, 승인형 자동화는 속도를 늦추는 것이 아니라 리스크를 줄이는 설계다.
다섯째, 좋은 에이전트는 “똑똑한 에이전트”보다 **운영 가능한 에이전트**다.

또 하나 중요한 점은, NemoClaw의 credential 저장 방식입니다. 현재 공식 문서 기준으로 호스트 측 운영용 credential은 `~/.nemoclaw/` 아래에 저장되며, NemoClaw 자체가 이를 at-rest 암호화하지는 않고 로컬 파일 소유권과 Unix 권한에 의존합니다. 따라서 운영 교육에서는 “도구가 다 막아준다”가 아니라, **호스트 보안과 파일 권한 관리도 운영 책임의 일부**라는 점을 분명히 짚어야 합니다. ([NVIDIA Docs][6])

## 14. 오늘의 핵심 정리

오늘 배운 핵심은 다음과 같습니다.

실행형 에이전트는 잘 만드는 것보다 **오래 안전하게 운영하는 것**이 더 중요합니다.
NemoClaw/OpenShell 구조는 기본 차단, 샌드박스, 정책 집행, 추론 라우팅, 자격증명 분리를 통해 그 운영 기반을 제공합니다. ([NVIDIA Docs][7])

좋은 자동화는 무인 자동화가 아니라 검토 가능한 자동화입니다.
따라서 업무는 자동 허용, 승인형, 금지 작업으로 나누어야 합니다.

읽기 중심 자동화는 넓게 시작할 수 있지만,
쓰기 중심 자동화는 좁게 설계하고 승인 단계를 둬야 합니다.

운영 정책 문서, 점검 체크리스트, 로그 검토 습관이 있어야
에이전트가 일회성 데모가 아니라 실제 운영 자산이 됩니다.

## 15. 4차시 마무리 메시지

NemoClaw와 OpenClaw를 함께 배운다는 것은
단순히 AI 도구를 배우는 것이 아닙니다.

그것은
**사용자의 디지털 환경에 연결된 실행형 AI를, 안전하고 책임 있게 운영하는 법을 배우는 것**입니다.

결국 중요한 질문은 이것입니다.

* 이 에이전트가 뭘 할 수 있나?
  보다
* 이 에이전트가 **어디까지 하도록 허용할 것인가?**

이 질문에 답할 수 있어야
개인 비서, 팀 비서, 전문가 에이전트, 디지털 워커로 확장해도 흔들리지 않습니다.
