# 2차시. 설치보다 중요한 운영 구조 이해

## 학습 목표
- NemoClaw의 구조를 설명할 수 있다.
- OpenShell sandbox, CLI plugin, blueprint, inference routing 개념을 이해한다.
- 왜 “로컬 실행”과 “안전한 실행”이 다른 문제인지 이해한다.

## 핵심 메시지
- NemoClaw의 핵심은 예쁜 UI가 아니라 실행 경로를 통제하는 운영 구조입니다.
- 공식 문서 기준 NemoClaw는 lightweight CLI plugin + versioned blueprint 구조로 OpenClaw를 통제된 샌드박스로 옮기는 방식입니다.

### 0~10분: 지난 시간 리뷰
- alpha/early preview 상태라는 점 확인
  
  NemoClaw는 2026년 3월 16일 GTC에서 발표된 직후 early preview로 공개되었습니다. 
  
  | 항목 | 현재 상태 |
  |------|----------|
  | **릴리스 단계** | Early Preview (2026년 3월 16일~) |
  | **프로덕션 준비** | ❌ 아님. 인터페이스, API, 동작이 예고 없이 변경될 수 있음 |
  | **공개 목적** | 피드백 수집 및 초기 실험 활성화 |
  | **커뮤니티 참여** | Issue, Discussion을 통한 피드백 환영 |
  
> "NemoClaw는 아직 프로덕션용이 아닙니다. 그러나 운영 구조를 이해하는 것은 PoC든 프로덕션이든 동일하게 중요합니다. 오늘은 '어떻게 설치하느냐'가 아니라 '설치 이후 무엇이 어떻게 돌아가느냐'를 이해하는 시간입니다."
- 지난 시간 핵심 복습:
  - **OpenClaw** = 실행형 AI 에이전트 플랫폼 (파일, 셸, 이메일, API를 직접 조작)
  - **NemoClaw** = OpenClaw를 OpenShell 샌드박스 안에서 안전하게 돌리기 위한 스택
  - 보안 철학 = **기본 차단, 명시적 허용 (deny by default, allow by policy)**


### 10~25분: 구조 설명
이 섹션의 목표는 3개의 레이어(OpenClaw / OpenShell / NemoClaw)가 각각 무엇을 담당하는지를 명확히 분리하는 것입니다.

#### ① OpenClaw: 에이전트/대화/도구/메모리

OpenClaw는 에이전트 그 자체입니다. NemoClaw가 없어도 독립적으로 동작합니다.

| 기능 | 설명 |
|------|------|
| **에이전트 런타임** | LLM과 도구를 연결해 자율적으로 작업 수행 |
| **대화 채널** | WhatsApp, Telegram, Slack, Discord 등 20여 개 메시징 플랫폼 |
| **도구 실행** | 셸 명령, 파일 조작, 웹 브라우징, API 호출 |
| **메모리** | 영속적 메모리(persistent memory)로 세션 간 컨텍스트 유지 |
| **스킬(Skills)** | 플러그인 방식으로 기능 확장 (100개 이상 빌트인 스킬) |

핵심 문제: OpenClaw 자체에는 **실행 범위를 제한하는 메커니즘이 부족**합니다. 에이전트가 호스트 시스템의 권한을 그대로 상속받으므로, 파일 삭제, 자격증명 접근, 임의 네트워크 요청이 모두 가능합니다.


#### ② OpenShell: 샌드박스와 정책 집행

OpenShell은 NVIDIA가 만든 오픈소스 런타임으로, 자율 에이전트가 더 안전하게 동작할 수 있는 격리된 실행 환경을 제공합니다. NemoClaw 전용이 아니라, 모든 에이전트에 사용할 수 있는 범용 보안 런타임입니다.

| 구성요소 | 역할 |
|---------|------|
| **샌드박스 컨테이너** | 에이전트를 격리된 환경에서 실행. Landlock(파일시스템) + seccomp(시스템 콜) + netns(네트워크 네임스페이스) 적용 |
| **게이트웨이** | Docker 컨테이너 내부의 K3s 클러스터로 샌드박스 수명주기 관리 |
| **정책 엔진** | 선언적 YAML 정책으로 4개 도메인(파일시스템, 네트워크, 프로세스, 추론) 통제 |
| **TUI** | 터미널 대시보드로 실시간 모니터링, 네트워크 요청 승인/거부 |
| **프로바이더 관리** | 에이전트 자격증명(API 키 등)을 환경변수로 주입. 파일시스템에 노출되지 않음 |

OpenShell의 4가지 정책 도메인:

| 정책 도메인 | 통제 대상 | 변경 시점 |
| :--- | :--- | :--- |
| **파일시스템** | 허용된 경로 외 읽기/쓰기 차단. <br>/sandbox, /tmp만 쓰기 가능 | 샌드박스 생성 시 잠금 |
| **네트워크** | 미인가 아웃바운드 연결 차단. <br>허용 목록 기반 | 런타임에 핫 리로드 가능 |
| **프로세스** | 권한 상승, 위험한 시스템 콜 차단 | 샌드박스 생성 시 잠금 |
| **추론** | 모델 API 호출을 통제된 백엔드로 리라우팅 | 런타임에 핫 리로드 가능 |


#### ③ NemoClaw: 둘을 연결하는 실행 계층

NemoClaw는 OpenClaw와 OpenShell을 연결하는 접착제 역할입니다. 그 자체가 에이전트도 아니고 샌드박스도 아닙니다.

공식 문서의 표현: "NemoClaw combines a lightweight CLI plugin with a versioned blueprint to move OpenClaw into a controlled sandbox."

**NemoClaw = Plugin + Blueprint**

| 구성요소 | 기술 스택 | 역할 |
|---------|----------|------|
| **Plugin** | TypeScript 패키지 | OpenClaw 게이트웨이 내부에서 실행. 추론 프로바이더 등록, `/nemoclaw` 슬래시 명령 처리, 사용자 상호작용 담당 |
| **Blueprint** | Python 아티팩트 (버전 관리) | 샌드박스 생성, 정책 적용, 추론 설정 등 모든 오케스트레이션 로직 담당. Plugin이 subprocess로 실행 |

**이렇게 분리한 이유 (설계 원칙):**

| 설계 원칙 | 설명 |
|----------|------|
| **Thin plugin, versioned blueprint** | 플러그인은 작고 안정적으로 유지. 오케스트레이션 로직은 블루프린트에 두고 독립적인 릴리스 주기로 발전 |
| **CLI 경계 존중** | `nemoclaw` CLI가 샌드박스 관리의 유일한 인터페이스 |
| **공급망 안전** | 블루프린트 아티팩트는 불변(immutable)이며, 버전 관리되고, 실행 전 다이제스트 검증을 거침 |
| **재현 가능한 설정** | setup을 다시 실행하면 동일한 블루프린트와 정책으로 샌드박스를 재생성 |

**Plugin 디렉토리 구조:**
```
nemoclaw/
├── src/
│   ├── index.ts                    플러그인 진입점 — 모든 명령 등록
│   ├── cli.ts                      Commander.js 서브커맨드 연결
│   ├── commands/
│   │   ├── launch.ts               OpenShell에 신규 설치
│   │   ├── connect.ts              샌드박스 대화형 셸 접속
│   │   ├── status.ts               블루프린트 실행 상태 + 샌드박스 헬스
│   │   ├── logs.ts                 블루프린트 및 샌드박스 로그 스트리밍
│   │   └── slash.ts                /nemoclaw 채팅 명령 핸들러
│   └── blueprint/
│       ├── resolve.ts              버전 해석, 캐시 관리
│       ├── fetch.ts                OCI 레지스트리에서 블루프린트 다운로드
│       ├── verify.ts               다이제스트 검증, 호환성 체크
│       ├── exec.ts                 블루프린트 runner를 subprocess로 실행
│       └── state.ts                영속 상태 (run ID 등)
├── openclaw.plugin.json            플러그인 매니페스트
└── package.json                    openclaw.extensions에 선언된 명령
```

**Blueprint 디렉토리 구조:**
```
nemoclaw-blueprint/
├── blueprint.yaml                  매니페스트 — 버전, 프로필, 호환성 정보
├── orchestrator/
│   └── runner.py                   CLI runner — plan / apply / status
├── policies/
│   └── openclaw-sandbox.yaml       기본 네트워크 + 파일시스템 정책 (엄격한 baseline)
```

**Blueprint 수명주기 5단계:**
```
[Resolve] → [Verify digest] → [Plan] → [Apply] → [Status]
```

| 단계 | 설명 |
|------|------|
| **1. Resolve** | 블루프린트 아티팩트를 찾고, `blueprint.yaml`의 `min_openshell_version`, `min_openclaw_version` 제약 조건을 확인 |
| **2. Verify** | 아티팩트 다이제스트를 기대값과 비교 검증 |
| **3. Plan** | 생성/업데이트할 OpenShell 리소스 결정 (게이트웨이, 프로바이더, 샌드박스, 추론 경로, 정책) |
| **4. Apply** | `openshell` CLI 명령을 호출하여 계획 실행 |
| **5. Status** | 현재 상태 보고 |

> "NemoClaw의 구조를 요리에 비유하면 이렇습니다. OpenClaw는 요리사(일을 하는 주체), OpenShell은 주방(격리된 작업 공간과 규칙), NemoClaw는 주방장(요리사를 주방에 배치하고 레시피를 전달하는 관리자)입니다."

### 25~40분: inference routing 개념
#### "에이전트가 모델에 직접 붙는 것" vs "런타임이 라우팅하는 것"

**일반적인 구조 (OpenClaw 단독):**
```
에이전트 ──직접──▶ 외부 LLM API (OpenAI, Anthropic 등)
```

이 구조에서는 에이전트가 어떤 모델을 호출하든, 어떤 데이터를 보내든 통제할 수 없습니다. API 키가 에이전트에 직접 노출되고, 요청 내용을 감사(audit)할 방법이 없습니다.

**NemoClaw 구조 (OpenShell 경유):**
```
에이전트 (샌드박스 내부) ──▶ OpenShell 게이트웨이 ──▶ 설정된 추론 프로바이더
```

에이전트의 추론 요청은 샌드박스를 직접 빠져나가지 않습니다. OpenShell이 모든 추론 호출을 가로채서 설정된 프로바이더로 라우팅합니다.

**이 구조의 의미:**

| 특성 | 직접 연결 | OpenShell 경유 |
| :--- | :--- | :--- |
| **API 키 노출** | 에이전트가 직접 보유 | 환경변수로 주입, <br>파일시스템에 노출 안 됨 |
| **모델 변경** | 코드 수정 후 재시작 필요 | `inference set` 명령으로 <br>런타임 중 즉시 전환 |
| **요청 감사** | 별도 로깅 구현 필요 | OpenShell TUI에서 실시간 모니터링 |
| **프라이버시** | 모든 데이터가 외부로 전송 | 프라우버시 라우터로 <br>로컬/클라우드 하이브리드 분기 |
| **비용 통제** | 에이전트가 무제한 호출 가능 | 추론 정책으로 프로바이더/모델 제한 |

#### 사용 가능한 추론 프로바이더 3가지

| 프로바이더 | 명령어 | 설명 |
|-----------|--------|------|
| **NVIDIA Cloud** | `openshell inference set --provider nvidia-nim --model <모델ID>` | build.nvidia.com을 통한 클라우드 추론. `NVIDIA_API_KEY` 필요 |
| **로컬 vLLM** | `openshell inference set --provider vllm-local --model <모델ID>` | 호스트에서 vLLM 서버를 직접 실행. 오프라인 개발에 적합 |
| **로컬 NIM** | `openshell inference set --provider nim-local --model <모델ID>` | 네트워크 내 NIM 컨테이너 사용. `NIM_API_KEY` 필요 |

#### "로컬 실행"과 "안전한 실행"이 다른 문제인 이유

> **질문:** "모델을 내 컴퓨터에서 로컬로 돌리면 안전한 거 아닌가요?"

| 오해 | 현실 |
|------|------|
| "로컬이면 데이터가 외부로 안 나간다" | 에이전트가 네트워크 요청을 보내면 로컬 모델이든 클라우드 모델이든 데이터는 나갈 수 있음 |
| "로컬이면 해킹 위험이 없다" | 에이전트가 호스트 시스템 권한을 그대로 가지므로, 악성 스킬이나 프롬프트 인젝션에 의한 피해는 동일 |
| "로컬이면 비용이 안 든다" | GPU 비용, 전력, 유지보수 비용 발생. 통제 없이 무한 추론 루프에 빠질 수도 있음 |
| "로컬이면 정책이 필요 없다" | 에이전트의 행동 범위를 제한하는 것은 모델 위치와 무관한 별개 문제 |

> ""로컬 실행"은 데이터가 어디서 처리되느냐의 문제이고, "안전한 실행"은 에이전트가 무엇을 할 수 있느냐의 문제입니다. NemoClaw는 후자를 해결합니다."


### 40~50분: fresh install / quickstart 관점 설명
#### 왜 기존 환경에 덮어씌우기보다 별도 테스트 환경이 좋은가

공식 문서의 명시적 안내: "NemoClaw currently requires a fresh installation of OpenClaw."

| 이유 | 설명 |
|------|------|
| **alpha 소프트웨어** | 인터페이스, API, 동작이 예고 없이 변경될 수 있음. 기존 워크플로우가 깨질 위험 |
| **샌드박스 격리 전제** | NemoClaw는 OpenClaw를 샌드박스 컨테이너 안에서 실행하는 구조. 기존 설치와 충돌 가능 |
| **정책 baseline** | 기본 정책이 모든 이그레스를 차단하므로, 기존에 동작하던 스킬이 정책에 의해 차단될 수 있음 |
| **재현 가능성** | setup을 다시 실행하면 동일한 블루프린트와 정책으로 샌드박스를 재생성하는 구조이므로, 클린 상태에서 시작하는 것이 안전 |

#### 권장 접근법
```
[기존 OpenClaw 환경]          [NemoClaw 테스트 환경]
├── 운영 중인 에이전트          ├── 별도 머신 또는 VM
├── 기존 스킬/메모리            ├── fresh install
├── 건드리지 않음              ├── 정책 실험 → 검증
                              └── 검증 완료 후 마이그레이션 검토
```

> "NemoClaw를 실험할 때 가장 좋은 전략은, 기존 환경을 건드리지 않고 별도의 클린 환경에서 시작하는 것입니다. alpha 소프트웨어에서는 '되돌릴 수 있는가'가 가장 중요한 판단 기준입니다."

**① 가장 바깥 — OpenShell이 제공하는 샌드박스 환경**

그림 맨 위의 자물쇠 아이콘과 함께 표시된 **"OpenShell provide sandbox env"**가 전체 구조를 감싸고 있습니다. NVIDIA OpenShell은 Docker 컨테이너 내부에서 Landlock(파일시스템 격리), seccomp(시스템 콜 필터링), netns(네트워크 네임스페이스)를 사용해 에이전트의 실행 환경을 격리합니다. 즉, 이 샌드박스 안에 있는 모든 것은 호스트 시스템과 분리된 통제된 공간에서 동작합니다.

**② 샌드박스 내부 상단 — OpenClaw Gateway**

샌드박스 안에서 가장 먼저 보이는 것이 **OpenClaw Gateway**입니다. 이것은 OpenClaw 에이전트의 핵심 제어 평면(control plane)으로, 메시징 채널로부터 들어오는 요청을 받아 에이전트에 전달하고, 에이전트의 응답과 도구 실행을 관리하는 역할을 합니다.

**③ 샌드박스 내부 좌측 — Policy Engine**

**Policy Engine**은 NemoClaw가 일반 OpenClaw와 가장 크게 차별화되는 핵심 요소입니다. 그림에 표시된 내용을 정리하면:

- **YAML로 작성된 규칙의 집합:** 정책은 선언적 YAML 파일(`openclaw-sandbox.yaml`)로 정의됩니다
- **기본 규칙 — 네트워크:** 명시적으로 허용 목록에 등록된 엔드포인트만 접근 가능 (나머지는 모두 차단)
- **기본 규칙 — 파일시스템:** 에이전트는 `/sandbox`와 `/tmp`에만 쓰기 가능 (나머지 시스템 경로는 읽기 전용)

이 정책 엔진이 에이전트의 모든 네트워크 요청과 파일 접근을 실시간으로 감시하고, 규칙에 따라 허용하거나 차단합니다.

**④ 샌드박스 내부 중앙 — AI Models Supported**

**AI Models Supported** 영역에는 NemoClaw에서 사용 가능한 모델 목록이 표시되어 있습니다:

1. Nemotron 3 Super 120B — NVIDIA의 기본 모델
2. Kimi K2.5
3. GLM-5
4. MiniMax M2.5
5. Qwen 3.5 397B
6. GPT-OSS 120B

NemoClaw는 특정 모델에 종속되지 않으며, NVIDIA Cloud(build.nvidia.com)를 통해 다양한 오픈 모델을 사용할 수 있습니다. 런타임 중 모델 전환도 가능합니다.

**⑤ 샌드박스 내부 우측 — OpenShell Gateway**

**OpenShell Gateway**는 추론 라우팅의 핵심입니다. 그림에 "Intercept inference request and route to configured provider"라고 설명되어 있듯이, 에이전트의 모든 추론 요청을 가로채서 설정된 프로바이더(NVIDIA Cloud, 로컬 vLLM, 로컬 NIM 등)로 라우팅합니다. 에이전트는 모델에 직접 접근하지 않고, 반드시 이 게이트웨이를 경유합니다. 이를 통해:

- API 키가 에이전트에 직접 노출되지 않음
- 모든 추론 호출을 감사(audit)할 수 있음
- 런타임 중 프로바이더/모델을 교체할 수 있음

그림 우측의 구름 모양 **Nvidia Cloud**는 OpenShell Gateway가 외부 클라우드 추론 서비스(build.nvidia.com)로 요청을 전달하는 경로를 나타냅니다.

**⑥ 샌드박스 외부 좌측 — NemoClaw CLI & NemoClaw Plugin**

샌드박스 바깥 좌측에 표시된 두 요소는 운영자(사람)가 샌드박스와 상호작용하는 진입점입니다:

- **NemoClaw CLI:** 샌드박스의 진입점(Entry point). `nemoclaw onboard`, `nemoclaw connect`, `nemoclaw status` 등의 명령으로 샌드박스를 생성·접속·관리합니다.
- **NemoClaw Plugin:** NemoClaw CLI를 구동하는 TypeScript 패키지. 내부적으로 Blueprint(Python 아티팩트)를 다운로드·검증·실행하여 샌드박스 환경을 오케스트레이션합니다.

그림에서 파란색 화살표가 CLI에서 샌드박스 안으로 향하는 것은, 운영자가 CLI를 통해 샌드박스 내부의 에이전트에 접근하는 흐름을 나타냅니다.

**⑦ 맨 아래 — Hardware Agnostic**

그림 하단의 **"Hardware Agnostic: run on Nvidia, AMD, Intel and others"**는 NemoClaw와 OpenShell이 NVIDIA GPU에 종속되지 않고 다양한 하드웨어에서 실행 가능하다는 것을 보여줍니다. 다만 GPU가 있으면 로컬 추론 성능이 향상되며, NVIDIA GPU 환경에서 최적의 성능을 발휘합니다.

---

**전체 데이터 흐름 요약:**
```
운영자 → [NemoClaw CLI] → [샌드박스 진입]
                              ↓
           [OpenClaw Gateway] — 에이전트가 작업 수행
                              ↓
           [Policy Engine] — 네트워크/파일시스템 요청을 규칙에 따라 허용 또는 차단
                              ↓
           [OpenShell Gateway] — 추론 요청을 가로채서 설정된 모델로 라우팅
                              ↓
           [Nvidia Cloud 또는 로컬 모델] — 실제 추론 수행 후 결과 반환
```

> "이 아키텍처의 설계 철학은 **"에이전트는 강력하게, 환경은 엄격하게"**입니다. OpenClaw 에이전트의 기능은 그대로 유지하면서, OpenShell 샌드박스와 Policy Engine이 실행 경로의 모든 지점에서 통제를 가합니다. 에이전트가 "무엇을 할 수 있는가"는 OpenClaw가 결정하고, "무엇을 해도 되는가"는 NemoClaw의 정책이 결정합니다."

## 실습
“내 NemoClaw 운영 구조도” 그리기:

| 요소 | 내가 채울 내용 | 예시 |
|------|---------------|------|
| **입력 채널** | 내가 사용할 메시징 플랫폼 | Telegram, Slack |
| **메모리/세션** | 에이전트가 기억해야 할 컨텍스트 | 프로젝트 컨텍스트, 팀원 선호도 |
| **스킬** | 에이전트에 연결할 도구/기능 | GitHub, Google Calendar, 이메일 |
| **외부 API** | 에이전트가 접근해야 할 외부 서비스 | api.github.com, api.slack.com |
| **승인/차단 지점** | 자동 허용할 것과 승인이 필요한 것 | 읽기는 자동, 쓰기/발송은 승인 |

## 산출물
- 운영 구조도 1장
- 내가 열어야 할 네트워크/닫아야 할 네트워크 목록

## 과제
사용 예정 도구를 분류:
1. 반드시 필요한 도구
2. 있으면 좋은 도구
3. 연결하면 위험한 도구

예시)
| 분류 | 기준 | 예시 |
|------|------|------|
| 🟢 **반드시 필요한 도구** | 이것 없이는 에이전트를 쓸 이유가 없음 | Google Calendar API, 사내 메신저 |
| 🟡 **있으면 좋은 도구** | 생산성은 높이지만 없어도 핵심 기능은 동작 | 뉴스 API, 날씨 API, 번역 API |
| 🔴 **연결하면 위험한 도구** | 에이전트에 권한을 주면 오남용·사고 위험이 높음 | 결제 API, DB 직접 접근, 루트 셸 |
