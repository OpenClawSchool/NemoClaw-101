# 2차시. 운영 구조 및 샌드박스

## OpenShell sandbox, 정책, 네트워크 통제로 이해하는 안전한 실행형 에이전트 구조

## 1. 왜 운영 구조를 먼저 이해해야 하는가

1차시에서 우리는 두 가지 핵심을 배웠습니다.

1. 실행형 에이전트는 일반 챗봇과 달리 파일·네트워크·API에 **직접** 접근할 수 있다.
2. 그래서 기능보다 보안이 먼저다.

그런데 "보안이 먼저"라는 말만으로는 실제로 무엇을 해야 하는지 알 수 없습니다.

2차시에서는 이 질문에 답합니다: **그 보안이 구체적으로 어떤 구조로 작동하는가?**

비유하자면 이렇습니다.

> 1차시에서는 "이 건물에는 보안이 필요하다"는 것을 알았습니다.
> 2차시에서는 "출입 게이트가 어디 있고, CCTV는 어디를 비추고, 열쇠는 누가 관리하는지"를 배웁니다.

---

## 2. 이 차시에서 배울 것

이번 차시를 마치면 다음 네 가지를 설명할 수 있어야 합니다.

| 순서 | 학습 목표 | 한 줄 설명 |
|------|-----------|------------|
| ① | **운영 구조 큰 그림** | 사용자 → OpenClaw → NemoClaw → OpenShell → 샌드박스, 이 흐름이 왜 이런 순서인지 |
| ② | **샌드박스가 필요한 이유** | 에이전트를 격리하면 무엇이 달라지는지 |
| ③ | **기본 차단 → 정책 허용 → 승인 흐름** | 네트워크 통제의 세 단계가 어떻게 연결되는지 |
| ④ | **통제 가능한 자동화** | "많이 자동화"가 아니라 "안전하게 자동화"하는 설계 원칙 |

각 절이 이 네 가지 중 하나에 대응합니다. 순서대로 따라가면 자연스럽게 연결됩니다. ([NVIDIA Docs][2])

## 3. NemoClaw 운영 구조 한눈에 보기

1차시에서 NemoClaw의 아키텍처 4계층(Plugin → Blueprint → Sandbox → Inference)을 배웠습니다. 이제 실제로 사용자의 요청이 어떤 경로를 따라 처리되는지 단계별로 따라가 봅시다.

**Step 1.** 사용자가 메시징 앱(Slack, WhatsApp 등)으로 에이전트에게 말을 겁니다.

**Step 2.** OpenClaw가 이 메시지를 받아 에이전트 로직을 실행합니다.

**Step 3.** 이때 에이전트는 내 컴퓨터 전체에서 돌아가는 것이 아닙니다. NemoClaw가 미리 만들어 둔 **OpenShell 샌드박스** 안에서 실행됩니다.

**Step 4.** 에이전트가 외부 API를 호출하거나 AI 모델에 질문해야 할 때, 직접 인터넷에 나가지 않습니다. OpenShell이 **정책을 확인한 뒤** 허용된 요청만 통과시킵니다.

```
사용자 → OpenClaw → [샌드박스 안에서 에이전트 실행] → OpenShell 정책 확인 → 허용된 외부 연결만 통과
```

이 흐름의 핵심은 간단합니다: **에이전트는 항상 통제된 경계 안에서 실행된다.** ([NVIDIA Docs][2])

## 4. OpenShell은 실무에서 어떻게 작동하나

1차시에서 OpenShell의 4가지 정책 도메인(파일시스템·네트워크·프로세스·추론)을 배웠습니다. 이것을 일상 비유로 바꿔보면 OpenShell이 실제로 무엇을 하는지 바로 이해됩니다.

| 정책 도메인 | 일상 비유 | 실무에서 의미하는 것 |
|-------------|-----------|----------------------|
| **파일시스템** | 사무실 캐비닛 열쇠 | 에이전트가 읽고 쓸 수 있는 폴더를 `/sandbox`와 `/tmp`로 제한 |
| **네트워크** | 건물 출입 게이트 | 허용된 외부 주소(API)에만 연결 가능, 나머지는 차단 |
| **프로세스** | 직원 권한 등급 | 위험한 시스템 명령(권한 상승 등) 실행 불가 |
| **추론** | 전화 교환대 | AI 모델 호출이 직접 외부로 나가지 않고 OpenShell을 거쳐 라우팅 |

정리하면, OpenShell은 단순한 컨테이너가 아닙니다.
**"에이전트가 할 수 있는 모든 행동에 규칙을 적용하는 보안 관제 시스템"**입니다. ([NVIDIA Docs][3], [4])

## 5. 샌드박스란 무엇인가

샌드박스를 한 문장으로 설명하면:

> **에이전트를 격리된 방 안에 넣어서 실행하는 구조.**

"그게 왜 중요하죠?" 이렇게 비교하면 바로 답이 나옵니다.

| | 샌드박스 **없이** 실행 | 샌드박스 **안에서** 실행 |
|---|---|---|
| 파일 접근 | 내 컴퓨터 전체 파일에 닿을 수 있음 | `/sandbox`와 `/tmp`만 읽기/쓰기 가능 |
| 네트워크 | 인터넷 어디든 자유롭게 접속 | 허용된 주소만 접속 가능 |
| API 키 | 저장된 키를 직접 가져다 쓸 수 있음 | API 키는 호스트에 남아 있고 샌드박스에는 전달되지 않음 |
| 사고 범위 | 잘못되면 시스템 전체에 영향 | 잘못되어도 샌드박스 안으로 제한 |

기술적으로는, OpenShell 샌드박스는 전용 Linux 네트워크 네임스페이스 안에서 동작합니다. 모든 트래픽이 호스트 측 프록시 경로를 통해서만 흐르기 때문에, 에이전트가 설정을 우회하더라도 샌드박스 밖으로 임의 통신하기 어렵습니다. ([NVIDIA Docs][1], [5])

중요한 포인트:
**샌드박스는 "보안을 완성"하는 장치가 아니라, "통제를 시작할 수 있게 만드는 최소 단위"**입니다.

## 6. 정책 기반 실행이 왜 중요한가

많은 초보자가 이렇게 생각합니다: "프롬프트에 '하지 마'라고 쓰면 되지 않나?"

결론부터 말하면, **안 됩니다.** 그 이유를 단계별로 살펴보겠습니다.

**1단계: 프롬프트가 할 수 있는 것**
프롬프트는 에이전트에게 "이렇게 해라", "이건 하지 마라"고 **말로** 전달합니다.

**2단계: 프롬프트가 할 수 없는 것**
운영체제 수준에서 네트워크를 막거나, 특정 프로그램의 실행 권한을 제한하거나, API 키 전달 방식을 바꾸는 것은 프롬프트로 **불가능**합니다.

**3단계: 그래서 정책이 필요하다**
런타임 정책은 에이전트가 말을 안 들어도 **물리적으로** 행동을 막습니다. 마치 열쇠가 없으면 문을 열 수 없는 것처럼.

> 프롬프트 = "이 방에 들어가지 마"라고 **말하는** 것
> 정책 = 애초에 **문을 잠그는** 것

공식 문서에서도 초기 부팅부터 정책을 적용하고, deny 로그를 보며 필요한 것만 열라고 권장합니다. ([NVIDIA Docs][1])

실무에서는 정책을 **세 층으로 나누어** 설계하면 관리하기 쉽습니다.

| 정책 층 | 설명 | 예시 |
|---------|------|------|
| **사전 허용** | 반복적이고 위험이 낮은 읽기 작업 | 캘린더 조회, 문서 검색 |
| **실행 중 승인** | 외부 쓰기, 민감한 조회, 모호한 요청 | 이메일 발송, CRM 수정 |
| **절대 금지** | 자동 삭제, 승인 없는 발송 | 대량 파일 삭제, 결제 승인 |

이 구분을 하지 않으면, 자동화가 늘수록 편리함보다 사고 위험이 더 빠리 커집니다.

## 7. 기본 차단이 실무에서 의미하는 것

1차시에서 NemoClaw의 "기본 차단, 명시적 허용" 원칙을 배웠습니다. 이것이 실제로 어떻게 작동하는지 단계별로 살펴봅시다.

**Step 1. 샌드박스 시작 시 — 모든 외부 연결이 차단된 상태로 시작합니다.**
인터넷 어디도 안 됩니다. 백지 상태입니다.

**Step 2. 정책 파일에서 허용할 대상만 열어줍니다.**
예: `api.github.com:443` → 허용, `api.slack.com:443` → 허용, 나머지 → 전부 차단.

**Step 3. 허용 목록에 없는 연결은 자동 차단됩니다.**
에이전트가 예상 밖 사이트에 접속하려 해도 물리적으로 불가능합니다.

이 방식이 왜 중요할까요? 만약 에이전트가 인터넷을 자유롭게 쓸 수 있다면 이런 일이 생길 수 있습니다.

| 위험 상황 | 설명 |
|-----------|------|
| 의도치 않은 URL 접근 | 에이전트가 잘못된 링크를 따라가는 경우 |
| 검증되지 않은 외부 API 호출 | 프롬프트 인젝션으로 유도된 호출 |
| 민감 데이터 외부 전송 | 내부 문서가 외부 서버로 유출 |
| 악성 콘텐츠 반응 | 웹페이지 안의 숨겨진 명령을 실행 |

**핵심:** 실행형 AI에서는 "연결 가능"보다 **"정책상 허용된 대상에만 연결 가능"**이 훨씬 중요합니다. ([NVIDIA Docs][3])

## 8. 허용되지 않은 요청은 어떻게 처리되는가

에이전트가 정책에 없는 주소에 연결하려 하면 어떻게 될까요? 조용히 빠져나가는 것이 아닙니다. **단계별로 처리됩니다.**

**Step 1. 차단 발생**
OpenShell이 해당 요청을 차단합니다.

**Step 2. deny 로그 기록**
다음 정보가 구조화된 로그로 남습니다. ([NVIDIA Docs][6])
* 어디로 연결하려 했는지 (목적지 호스트, 포트)
* 어떤 프로그램이 시도했는지 (바이너리)
* 왜 거부되었는지 (거부 이유)

**Step 3. 운영자가 로그를 확인하고 판단**

예를 들어 수강생이 "왜 GitHub API가 안 되죠?"라고 묻는다면, 답은 이 순서를 따릅니다.

```
1. deny 로그에서 어떤 요청이 차단되었는지 확인
2. 정말 필요한 목적지인지 판단
3. 읽기 전용인지, 쓰기 요청인지 구분
4. 필요한 경우에만 최소 범위로 정책 추가
```

이 흐름 자체가 실행형 에이전트 운영의 **기본 습관**입니다.
핵심은 "감으로 고치는 것"이 아니라 **"로그를 보고 정책을 다듬는 것"**입니다.

## 9. L7 정책: 읽기와 쓰기를 나눠서 생각해야 한다

네트워크 통제를 "이 사이트 허용/차단" 수준으로만 이해하면 부족합니다.
왜냐하면 **같은 사이트라도 읽기는 안전하고 쓰기는 위험할 수 있기 때문**입니다.

예를 들어보겠습니다.

| 작업 | HTTP 메서드 | 위험도 | 권장 정책 |
|------|-------------|--------|----------|
| GitHub 이슈 조회 | GET | 낮음 | ✅ 사전 허용 |
| GitHub PR 생성 | POST | 높음 | ⚠️ 승인 후 허용 |
| 이메일 수신함 조회 | GET | 낮음 | ✅ 사전 허용 |
| 이메일 발송 | POST | 높음 | ⚠️ 승인 후 허용 |
| 고객 데이터 수정 | PUT | 매우 높음 | ⚠️ 승인 후 허용 |
| 대량 레코드 삭제 | DELETE | 위험 | 🚫 절대 금지 |

OpenShell은 호스트, 포트, HTTP 메서드, 심지어 **어떤 프로그램이 접근하는지**까지 정책으로 제한할 수 있습니다. `binaries` 필드를 통해 실행 파일을 식별하고, 바이너리 무결성이 변하면 차단할 수도 있습니다. ([NVIDIA Docs][6])

**핵심:** 자동화 설계에서 중요한 것은 "어느 사이트를 열 수 있나"가 아니라 **"어떤 행위까지 허용할 것인가"**입니다.

## 10. 추론 라우팅과 자격증명 분리가 실무에서 주는 이점

1차시에서 추론 계층의 기본 구조를 배웠습니다. 여기서는 이 구조가 왜 실무에서 유용한지를 구체적으로 설명합니다.

먼저, 보통의 방식과 NemoClaw 방식을 비교해보겠습니다.

| | 보통의 방식 | NemoClaw 방식 |
|---|---|---|
| API 키 위치 | 에이전트가 직접 갖고 있음 | 호스트에 보관, 샌드박스에는 전달 안 됨 |
| 모델 호출 | 에이전트가 직접 외부 API 호출 | `inference.local`로 요청 → OpenShell이 라우팅 |
| 모델 변경 | 에이전트 코드 수정 필요 | 에이전트 변경 없이 OpenShell에서 교체 |

이 구조가 실무에 주는 이점은 세 가지입니다.

1. **비밀키 격리** — 샌드박스가 해킹당해도 API 키는 안전합니다.
2. **모델 교체 용이** — 공급자를 바꿔도 에이전트는 그대로입니다.
3. **팀 거버넌스** — 키 관리를 호스트 계층에서 통합 관리할 수 있습니다. ([NVIDIA Docs][7])

> "에이전트는 모델을 직접 아는 것이 아니라, 정해진 내부 라우트로 요청을 보내고, 실제 비밀 정보와 외부 연결은 운영 계층이 맡는다."

## 11. 실전 시나리오로 이해하는 운영 구조

### 11.1 개인 생산성 비서

아침 브리핑, 캘린더 요약, 할 일 정리, 뉴스/API 조회 같은 작업은 비교적 읽기 중심입니다.
이 경우에는 읽기 전용 API와 일정 조회를 허용하고, 메시지 발송이나 외부 업로드는 승인형으로 두는 것이 적절합니다.

### 11.2 창업가 운영 비서

고객 문의 조회, CRM 확인, 일정 조율 초안 작성은 자동화하기 좋습니다.
반면 고객에게 실제 메일을 보내거나, CRM에 레코드를 수정하거나, 외부 서비스에 POST 요청을 보내는 것은 승인형이 바람직합니다.

### 11.3 전문가 에이전트

문서 검색, 요약, 강의자료 정리, 고객 맞춤 응답 초안 작성은 자동화 가치가 큽니다.
하지만 실제 발송, 계약 문서 외부 제출, 고객 데이터 업로드는 반드시 사람 검토가 필요합니다.

### 11.4 개발자/운영 에이전트

이슈 조회, 로그 조회, 문서 검색, 빌드 상태 확인은 자동화하기 좋습니다.
반면 PR 생성, `git push`, 배포, 인프라 수정은 승인형으로 설계해야 합니다.

이 사례들의 공통 원칙은 하나입니다.

**읽기 자동화는 넓게, 쓰기 자동화는 좁게.**
이 원칙이 지켜져야 실행형 AI가 실무에서 쓸 만해집니다.

## 12. 실습

### 실습 1. 내 업무를 읽기 / 쓰기로 분류하기

아래 세 단계를 따라 내 업무를 분류해보세요.

**Step 1.** 내 업무 중 자동으로 읽어와도 되는 것을 나열합니다.
**Step 2.** 실제 변경(생성, 수정, 삭제)이 일어나는 것을 나열합니다.
**Step 3.** 외부 전송이 포함된 업무를 나열합니다.

예시:

| 구분 | 업무 예시 |
|------|-----------|
| 읽기 | 캘린더 조회, 메일 요약, 문서 검색 |
| 쓰기 | 메일 발송, CRM 입력, 파일 삭제 |
| 외부 전송 | 고객 답장, API POST, 메시지 발신 |

### 실습 2. 허용 / 승인 / 금지 표 만들기

실습 1에서 분류한 업무를 아래 표에 배치합니다. 각 칸에 3개씩 채워보세요.

| ✅ 자동 허용 | ⚠️ 승인 후 허용 | 🚫 절대 금지 |
|-------------|---------------|-------------|
| 예: 일정 조회 | 예: 이메일 발송 | 예: 대량 삭제 |
| 예: 회의자료 요약 | 예: CRM 수정 | 예: 미검토 고객 발송 |
| 예: 내부 문서 검색 | 예: Git push | 예: 결제 승인 자동화 |

### 실습 3. 내 에이전트 운영 구조 그리기

아래 구조를 참고하여 다이어그램을 손으로 그리거나 문서로 작성합니다.

```
사용자
  ↓
OpenClaw (대화 인터페이스)
  ↓
NemoClaw (온보딩 · 수명주기 관리)
  ↓
┌─────────── OpenShell sandbox ───────────┐
│  에이전트 실행                            │
│    ↓                                     │
│  inference.local → [추론 라우팅]          │
│    ↓                                     │
│  정책 확인 → 허용? → 외부 연결            │
│             → 차단? → deny 로그           │
│             → 승인? → 운영자에게 알림      │
└──────────────────────────────────────────┘
  ↕ (host credentials는 샌드박스 밖에 보관)
```

이 실습의 목적은 구조를 외우는 것이 아니라,
**내가 맡길 업무가 어떤 경계를 통과하는지 시각적으로 이해하는 것**입니다.

### 실습 4. @userinfobot으로 텔레그램 숫자 ID 확인 및 NemoClaw 연동

@userinfobot은 텔레그램(Telegram)에서 사용자의 고유한 **숫자 ID(Numeric User ID)**를 확인하기 위해 사용하는 봇입니다.

NemoClaw 설정 과정 중 텔레그램 채널을 연동할 때 다음과 같은 용도로 사용됩니다.

#### 1. 사용 용도

- **고유 ID 확인:** 텔레그램 검색창에서 `@userinfobot`을 찾아 메시지를 보내면, 즉시 사용자 계정의 고유 숫자 ID를 알려줍니다. 이는 일반적인 사용자 이름(Username)과는 다른 고유 식별 번호입니다.

#### 2. NemoClaw에서의 중요성

- **접근 제어(Security):** NemoClaw 온보딩 과정에서 이 숫자 ID를 입력하면, 생성된 AI 에이전트(봇)가 해당 ID를 가진 사용자에게만 응답하도록 설정됩니다.
- **보안 강화:** 이를 통해 봇의 사용자 이름이 공개되더라도 타인이 사용자의 AI 에이전트를 가로채거나 오용하는 것을 방지할 수 있습니다.

#### 3. 설정 방법 (NemoClaw 대시보드)

- **허용 목록 등록:** 대시보드의 **Channels → Telegram** 설정 섹션 내 **Allow From** 필드에 `@userinfobot`을 통해 확인한 숫자 ID를 입력하고 저장(Apply)합니다.
- **DM 정책 설정:** `dmPolicy`를 `allowlist`로 설정하면 등록된 숫자 ID를 가진 사용자만 봇과 대화할 수 있습니다.

> **트러블슈팅:** 만약 NemoClaw 설정 후 봇이 응답하지 않는다면, `@userinfobot`으로 확인한 ID가 대시보드 설정의 `allowFrom` 항목에 정확히 입력되었는지 다시 한번 확인해야 합니다.

### 실습 5. Headless Server에서 NemoClaw 상태 확인 및 접속

모니터나 GUI가 없는 서버(Headless Server)에서 NemoClaw의 상태를 확인하고 접속하는 방법은 다음과 같습니다.

#### 1. CLI를 통한 상태 및 정보 확인

터미널(SSH 연결 상태)에서 다음 명령어를 사용하여 현재 시스템 및 샌드박스의 상태를 확인할 수 있습니다.

- **전체 상태 확인:** `nemoclaw status` 명령어를 입력하면 현재 실행 중인 샌드박스 목록과 상태를 볼 수 있습니다.
- **특정 샌드박스 확인:** `nemoclaw <샌드박스명> status`를 통해 특정 에이전트의 상세 정보를 확인할 수 있습니다.

#### 2. 샌드박스 접속 및 TUI 사용

CLI 환경에서 에이전트와 직접 대화하거나 관리하려면 샌드박스 내부로 접속해야 합니다.

1. **샌드박스 연결:** `nemoclaw <샌드박스명> connect` 명령어를 입력하여 샌드박스 쉘(Shell)로 들어갑니다.
2. **채팅 인터페이스(TUI) 실행:** 샌드박스 쉘 안에서 `openclui` 명령어를 입력하면 텍스트 기반의 채팅 인터페이스가 실행됩니다.

#### 3. 대시보드(웹 UI) 원격 접속

NemoClaw 대시보드는 기본적으로 서버의 `http://localhost:8080`에서 실행됩니다. Headless Server의 웹 화면을 로컬 PC 브라우저에서 보려면 **SSH 포트 포워딩**이 필요합니다.

- **포트 포워딩 방법:** 로컬 PC 터미널에서 서버에 접속할 때 다음과 같이 입력합니다.
    ```bash
    ssh -L 8080:localhost:8080 <사용자이름>@<서버IP>
    ```
- **브라우저 접속:** 위와 같이 연결한 후, 로컬 PC 브라우저 주소창에 `http://localhost:8080`을 입력하면 서버에서 돌아가는 대시보드에 접속할 수 있습니다.

#### 4. 주의사항: 환경 변수 로드

SSH 세션을 새로 열었을 때 `nemoclaw` 명령어를 찾을 수 없다면, 설치 경로가 현재 셸에 반영되지 않은 것입니다. 이 경우 다음 명령어를 실행하여 경로를 새로고침하세요.

```bash
source ~/.bashrc  # 또는 zsh 사용 시 source ~/.zshrc
```

## 14. 오늘의 핵심 정리

| # | 핵심 정리 |
