# 2차시. 설치보다 중요한 운영 구조 이해

## 학습 목표
- NemoClaw의 구조를 설명할 수 있다.
- OpenShell sandbox, CLI plugin, blueprint, inference routing 개념을 이해한다.
- 왜 “로컬 실행”과 “안전한 실행”이 다른 문제인지 이해한다.

## 핵심 메시지
- NemoClaw의 핵심은 예쁜 UI가 아니라 실행 경로를 통제하는 운영 구조입니다.
- 공식 문서 기준 NemoClaw는 lightweight CLI plugin + versioned blueprint 구조로 OpenClaw를 통제된 샌드박스로 옮기는 방식입니다.

## 60분 진행안

### 0~10분: 지난 시간 리뷰
- alpha/early preview 상태라는 점 확인
- 프로덕션용이 아니라 PoC·학습·검증 중심으로 접근해야 함 설명

### 10~25분: 구조 설명
- **OpenClaw**: 에이전트/대화/도구/메모리
- **OpenShell**: 샌드박스와 정책 집행
- **NemoClaw**: 둘을 연결하는 실행 계층

### 25~40분: inference routing 개념
- 로컬 모델 / 원격 모델 / 정책 경유 실행 이해
- “에이전트가 모델에 직접 붙는 것”과 “런타임이 라우팅하는 것”의 차이 설명
- 공식 문서에서는 NemoClaw가 open models와 OpenShell을 결합해 더 안전한 실행을 지향한다고 설명합니다.

### 40~50분: fresh install / quickstart 관점 설명
- 왜 기존 환경에 무리하게 덮어씌우기보다 별도 테스트 환경이 좋은지 설명
- Quickstart는 alpha 소프트웨어이며, 새로 시작하는 흐름을 전제로 안내하고 있습니다.

### 50~60분: 운영 아키텍처 그려보기
- 채널, 에이전트, 모델, 파일, 외부 네트워크, 승인 포인트

## 실습
“내 NemoClaw 운영 구조도” 그리기:
- 입력 채널
- 메모리/세션
- 스킬
- 외부 API
- 승인/차단 지점 표시

## 산출물
- 운영 구조도 1장
- 내가 열어야 할 네트워크/닫아야 할 네트워크 목록

## 과제
사용 예정 도구를 분류:
1. 반드시 필요한 도구
2. 있으면 좋은 도구
3. 연결하면 위험한 도구
