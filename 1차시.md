# 1차시. NemoClaw가 무엇이고 왜 필요한가

## 학습 목표

- OpenClaw와 NemoClaw의 관계를 설명할 수 있다.
- 일반 AI 챗봇과 실행형 에이전트의 차이를 이해한다.
- “왜 보안이 기능보다 먼저인가”를 이해한다.

## 핵심 메시지

- 일반 AI 챗봇은 질문에 답한다. OpenClaw는 질문에 답할 뿐 아니라 실행한다. NemoClaw는 그 실행을 더 통제된 경계 안에 넣는 도구다.
- 공식 문서 기준 NemoClaw는 OpenClaw 자체를 대체하는 제품이 아니라, NVIDIA OpenShell 기반 샌드박스에서 OpenClaw를 더 안전하게 돌리기 위한 플러그인/스택입니다.

## 60분 진행안

### 0~10분: 과정 소개

### OpenClaw와 NemoClaw의 한 줄 정의

| 구분               | 한 줄 정의                                                                                                                                                                                                                         |
| :----------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **OpenClaw** | LLM을 실제 소프트웨어와 연결해, 파일 읽기/쓰기, 셸 명령 실행, 웹 브라우징, 이메일 발송, API 제어 등을 자율적으로 수행하는 오픈소스 AI 에이전트 플랫폼. WhatsApp·Telegram·Slack·Discord 등 메시징 앱을 주 인터페이스로 사용한다. |
| **NemoClaw** | OpenClaw 상시 가동(always-on) 어시스턴트를 더 안전하게 실행할 수 있도록, NVIDIA OpenShell 런타임과 Nemotron 오픈소스 모델을 단일 명령으로 설치하는 오픈소스 레퍼런스 스택.                                                         |

> "OpenClaw는 여러분의 개인 AI 비서가 실제로 일을 해주는 플랫폼입니다. NemoClaw는 그 비서에게 출입증과 규칙을 부여하는 보안 시스템입니다."

### 왜 "기능 소개"보다 "운영 구조 이해"가 먼저인지

**문제 제기 — OpenClaw의 폭발적 성장과 보안 우려의 동시 발생:**

OpenClaw는 오스트리아 개발자 Peter Steinberger가 2025년 11월 처음 공개한 이후, 2026년 초 GitHub 별 145,000개 이상, 포크 20,000개 이상을 기록하며 역사상 가장 빠르게 성장한 오픈소스 프로젝트 중 하나가 되었습니다. 그러나 이 성장과 함께 심각한 보안 우려도 동반되었습니다. Gartner 분석가들은 OpenClaw의 설계가 "기본적으로 보안이 미흡하다(insecure by default)"고 평가했고, Cisco의 보안 분석가들은 "보안 악몽"이라고까지 표현했습니다. 실제로 2026년 2월 말 한 주간에만 40개 이상의 보안 패치가 배포되었으며, 악의적 웹사이트가 로컬 OpenClaw에 명령을 보낼 수 있는 심각한 "ClawJacked" 취약점이 발견되기도 했습니다.

**핵심 포인트:**

AI 에이전트가 내 시스템의 파일을 읽고, 명령을 실행하고, 외부 API를 호출할 수 있다면, 그 권한의 범위와 통제가 기능보다 먼저 정의되어야 합니다. 자율 AI 에이전트는 임의의 네트워크 요청을 보내고, 호스트 파일시스템에 접근하고, 모든 추론 엔드포인트를 호출할 수 있는데, 가드레일 없이는 보안·비용·컴플라이언스 리스크가 무인 운영 시간이 늘수록 커집니다.

"이 도구가 무엇을 할 수 있는가"보다 "이 도구를 어떻게 안전하게 운영할 것인가"를 먼저 이해해야, 나중에 배울 모든 기능이 올바른 맥락에 놓입니다. NemoClaw가 등장한 이유 자체가 이 문제의 답입니다.

> "여러분이 회사 컴퓨터에 새 인턴을 앉히는데, 첫날부터 모든 파일과 이메일에 접근 권한을 주겠습니까?"

### 10~25분: 일반 LLM 챗봇 vs OpenClaw vs NemoClaw 비교

### 대화형 인터페이스 + 도구 실행 + 메모리 + 자동화 + 멀티채널 구조 비교

| 구분                              | 일반 LLM 챗봇         | OpenClaw                                                                            | NemoClaw 환경의 OpenClaw                                                  |
| :-------------------------------- | :-------------------- | :---------------------------------------------------------------------------------- | :------------------------------------------------------------------------ |
| **대화 `<br>`인터페이스** | 웹/앱 채팅 창         | WhatsApp, Telegram, Slack,`<br>`Discord, Google Chat 등 20여 개 채널              | 동일 (샌드박스 내부에서 동작)                                             |
| **도구 실행**               | ❌ 불가 (또는 제한적) | ✅ 파일 읽기/쓰기, 셸 명령 실행,`<br>`웹 브라우징, 이메일/API 제어                | ✅ 동일 +**YAML 정책으로 `<br>`허용/차단을 세밀하게 통제**        |
| **메모리**                  | 세션 내 컨텍스트 유지 | ✅ 영속적 메모리로 몇 주에 걸친 `<br>`대화·습관을 기억하고 적응                  | ✅ 동일, 파일시스템 접근이 `<br>`/sandbox 및 /tmp로 제한 및 격리        |
| **자동화**                  | ❌ 수동 프롬프트 입력 | ✅ 예약 작업(cron), 웹훅 트리거를 `<br>`통한 24/7 자율 동작                       | ✅ 동일 +**네트워크 이그레스 `<br>`정책으로 자동 실행 범위 제한** |
| **보안/격리**               | 클라우드 사업자 관리  | ⚠️ 시스템 권한 상속으로 보안 위험 존재.`<br>`잘못 설정 시 데이터 유출/삭제 가능 | ✅ OpenShell이 커널 수준까지 `<br>`4개 도메인에 방어 심층 정책 적용     |
| **모델 선택**               | 공급자 고정           | 모델 무관 (BYO API key)                                                             | Nemotron 로컬 실행 +`<br>`프라이버시 라우터로 하이브리드 구성           |

### "답변형 AI"와 "실행형 AI"의 책임 차이

**답변형 AI의 책임 범위:** 사용자가 질문하면 텍스트로 답변합니다. 잘못된 정보를 주더라도 시스템에 직접적 피해는 없습니다. 판단과 실행은 사용자 몫입니다.

**실행형 AI의 책임 범위:** 일반 챗봇과 달리 OpenClaw는 지시에 따라 행동하고, 애플리케이션과 상호작용하며, 상시 감독 없이도 작업을 수행하는 자율 에이전트입니다. AI가 직접 파일을 삭제하고, 이메일을 발송하고, 코드를 실행합니다. 잘못된 판단이 곧 실제 피해로 이어집니다.

> **핵심 비유:** "일반 챗봇은 **조언자**입니다 — '이렇게 하세요'라고 말합니다. OpenClaw는 **실행자**입니다 — 실제로 합니다. NemoClaw는 그 실행자에게 **출입증과 규칙을 부여하는 보안 담당자**입니다."

**실행형 AI가 위험해지는 4가지 시나리오:**

| 위험 유형                  | 설명                                                                                 |
| -------------------------- | ------------------------------------------------------------------------------------ |
| **권한 초과**        | OpenClaw는 시스템 권한을 상속받으므로, 잘못 설정 시 파일 삭제·민감 데이터 유출 가능 |
| **악성 스킬**        | 커뮤니티 스킬(플러그인) 중 악성코드·데이터 탈취 루틴이 포함될 수 있음               |
| **프롬프트 인젝션**  | 에이전트가 처리하는 문서·이메일·웹페이지에 숨겨진 악의적 명령이 실행될 수 있음     |
| **의도치 않은 동작** | 선의의 명령도 잘못 해석되어 오발송·중요 파일 수정 등 원치 않는 결과 발생 가능       |

### NemoClaw 설치

1. 사전 준비

   ### 하드웨어 요구사항

   | 리소스                 | 최소       | 권장        |
   | :--------------------- | :--------- | :---------- |
   | **CPU**          | 4 vCPU     | 4 vCPU 이상 |
   | **RAM**          | 8 GB       | 16 GB       |
   | **디스크(Disk)** | 20 GB 여유 | 40 GB 여유  |


   > **⚠️ 메모리 관련 주의사항:** 설치 시 압축 해제 및 여러 데몬(Docker, k3s, OpenShell 등)이 동시 실행되며 메모리를 많이 사용합니다. RAM이 8GB 미만인 시스템에서는 **메모리 부족(OOM)으로 인한 프로세스 종료**를 방지하기 위해 반드시 **8GB 이상의 스왑(swap) 공간**을 구성해야 합니다. (단, 스왑 사용 시 성능 저하가 발생할 수 있습니다.)
   >

   ### 소프트웨어 요구사항

   | 의존성                  | 요구 버전 및 환경                                                                                           |
   | :---------------------- | :---------------------------------------------------------------------------------------------------------- |
   | **운영 체제(OS)** | Linux (Ubuntu 22.04 LTS+), macOS (Apple Silicon),`<br>`Windows WSL2 (Docker Desktop 백엔드), DGX Spark 등 |
   | **Node.js / npm** | Node.js 20+ / npm 10+                                                                                       |
   | **Docker**        | 설치 및 엔진 실행 중                                                                                        |
   | **OpenShell**     | [NVIDIA OpenShell](https://github.com/NVIDIA/OpenShell) 설치 완료                                              |

   ### NVIDIA API 키 발급 (추론용)

   NemoClaw 에이전트가 사용할 클라우드 모델(Nemotron 등) 추론을 위해 NVIDIA API Catalog 키가 필요합니다.

   1. **[build.nvidia.com](https://build.nvidia.com)**에 접속합니다.
   2. 목록에서 모델 하나를 선택합니다 (예: **Llama 3** 또는 **Nemotron 3 Super**).
   3. 모델 상세 페이지 우측 또는 API 섹션의 **[Get API Key]** 버튼을 클릭합니다.
   4. NVIDIA 계정으로 로그인합니다 (계정이 없는 경우 생성 필요).
   5. **[Generate Key]**를 클릭하여 생성된 키(`nvapi-`로 시작)를 복사하여 안전한 곳에 보관합니다.

   > **💡 팁:** 발급받은 키는 터미널 또는 환경 설정 파일(`.env`)에 `NVIDIA_API_KEY`라는 이름으로 등록해 두면 NemoClaw가 자동으로 인식합니다.
   >

   ### Docker 및 GPU 가속 환경 설정 (Jetson/Ubuntu 기준)

   NemoClaw 구동을 위해 필요한 Docker 설치 및 GPU 가속을 위한 NVIDIA Container Toolkit 설정 과정입니다.

   **1. 기본 필수 패키지 설치 및 기존 도커 제거**

   ```bash
   sudo apt-get update && sudo apt-get install -y curl
   for pkg in docker.io docker-doc docker-compose docker-compose-v2 podman-docker containerd runc; do sudo apt-get remove $pkg; done
   ```

   **2. 도커 공식 GPG 키 및 리포지토리(저장소) 추가**

   ```bash
   curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg

   echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/docker-archive-keyring.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
   ```

   **3. 도커 엔진 설치**

   ```bash
   sudo apt-get update && sudo apt-get install -y docker-ce docker-ce-cli containerd.io
   ```

   **4. 사용자 권한 설정 (중요)**
   NemoClaw 설치 스크립트는 `sudo` 권한 없이 실행되어야 하므로, 현재 사용자를 `docker` 그룹에 추가해야 합니다.

   ```bash
   sudo usermod -aG docker $USER
   ```

   **5. NVIDIA Container Toolkit 설치 및 설정**

   ```bash
   sudo apt-get install -y nvidia-container-toolkit
   sudo nvidia-ctk runtime configure --runtime=docker
   ```

   **6. NVIDIA를 기본 도커 런타임으로 설정**
   `sudo nano /etc/docker/daemon.json` 파일을 열어 다음 내용을 작성합니다.

   ```json
   {
       "default-runtime": "nvidia",
       "runtimes": {
           "nvidia": {
               "path": "nvidia-container-runtime",
               "runtimeArgs": []
           }
       }
   }
   ```

   **7. 도커 데몬 및 서비스 재시작**

   ```bash
   sudo systemctl daemon-reload
   sudo systemctl restart docker
   ```

   > **⚠️ 주의:** 설정을 마친 후 시스템에서 로그아웃했다가 다시 로그인하거나 재부팅해야 권한 설정이 완전히 적용됩니다.
   >

### 25~40분: NemoClaw의 공식 포지션

### OpenClaw plugin for NVIDIA OpenShell

**NemoClaw는 OpenClaw의 경쟁 제품이 아닙니다.** 처음에는 엔터프라이즈 경쟁 제품이라는 루머가 있었으나, 실제로는 OpenClaw를 NVIDIA의 오픈 에이전트 스택에 통합하는 프로젝트이며, Agent Toolkit이 보안과 가드레일이라는 빠진 조각을 채워주는 구조입니다.

2026년 3월 16일 GTC에서 발표된 NemoClaw는 "OpenClaw 에이전트 플랫폼을 위한 NVIDIA 스택"으로 공식 포지셔닝되었으며, 엔터프라이즈급 OpenClaw 배포판에 가까운 것으로 자리매김하고 있습니다.

NemoClaw는 NVIDIA가 빠르게 성장하는 OpenClaw 생태계에 기여하고자 내놓은 하나의 결과물입니다. 장기 실행 자율 에이전트를 위한 프레임웍이라고 하겠는데, 그 아이디어는 단순합니다: 명령어 하나로 OpenClaw와 Nemotron 모델, 그리고 NVIDIA의 새로운 보안 런타임 OpenShell을 함께 설치할 수 있습니다. 다시 말해서, NVIDIA는 이제 모델에 동력(Power)을 공급하는 데 그치는 플레이어가 아니라, 에이전트 그 자체의 기저에 자리를 잡는 플레이어가 되겠다는 겁니다.

**아키텍처 4계층:**

| 계층                     | 구성요소                  | 역할                                                                                                                   |
| ------------------------ | ------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Plugin 계층**    | nemoclaw CLI (TypeScript) | 추론 프로바이더와 /nemoclaw 슬래시 명령을 등록하는 경량 패키지. OpenClaw 게이트웨이 내부에서 실행                      |
| **Blueprint 계층** | Blueprint Runner (Python) | 샌드박스 생성, 정책 적용, 추론 설정 등 오케스트레이션 로직을 담당하는 버전 관리 아티팩트                               |
| **Sandbox 계층**   | OpenShell 컨테이너        | OpenClaw가 격리된 컨테이너 안에서 실행. 네트워크 이그레스는 정책으로 제한, 파일시스템은 /sandbox와 /tmp만 쓰기 가능    |
| **Inference 계층** | 추론 라우터               | 에이전트의 추론 요청은 샌드박스를 직접 빠져나가지 않음. OpenShell이 모든 추론 호출을 가로채 설정된 프로바이더로 라우팅 |

**Blueprint 수명주기 4단계:** Resolve(아티팩트 위치 확인) → Verify(다이제스트 검증) → Plan(리소스 계획) → Apply(OpenShell CLI 실행)

### Open Source Models 활용

NemoClaw는 특정 모델에 종속되지 않습니다. 모든 코딩 에이전트를 사용할 수 있으며, Nemotron을 포함한 오픈 모델을 로컬에서 실행하거나 프라이버시 라우터를 통해 클라우드 프론티어 모델에도 연결할 수 있습니다.

- **기본 모델:** NVIDIA Endpoints를 통한 Nemotron 3 Super 120B
- **로컬 대안:** Ollama 또는 vLLM을 통해 Qwen, DeepSeek 등 다른 오픈소스 모델 사용 가능
- **모델 교체:** 샌드박스를 재시작하지 않고도 런타임 중 모델 전환 가능
- **하이브리드 구성:** 민감한 컨텍스트는 로컬 모델에서 처리, 고성능이 필요한 작업은 클라우드로 분기

### Secure Environment / Sandbox / Policy 기반 실행

**OpenShell의 4가지 정책 도메인:**

OpenShell은 파일시스템·네트워크·프로세스·추론 4개 정책 도메인에 걸쳐 방어 심층(defense in depth)을 적용합니다. 정책은 선언적 YAML 파일로 정의됩니다.

| 정책 도메인          | 설명                                     | 변경 시점               |
| -------------------- | ---------------------------------------- | ----------------------- |
| **파일시스템** | 허용된 경로 외 읽기/쓰기 차단            | 샌드박스 생성 시 잠금   |
| **네트워크**   | 미인가 아웃바운드 연결 차단              | 런타임에 핫 리로드 가능 |
| **프로세스**   | 권한 상승 및 위험한 시스템 콜 차단       | 샌드박스 생성 시 잠금   |
| **추론**       | 모델 API 호출을 통제된 백엔드로 리라우팅 | 런타임에 핫 리로드 가능 |

**핵심 동작 원리:**

기본 정책(`openclaw-sandbox.yaml`)은 명시적으로 나열된 엔드포인트를 제외한 모든 네트워크 이그레스를 차단합니다. 에이전트가 정책에 없는 엔드포인트에 접근을 시도하면, OpenShell이 해당 요청을 차단하고 TUI(터미널 UI)에 표시하여 운영자가 실시간으로 승인 또는 거부할 수 있습니다.

**정책 커스터마이징 예시:**

```yaml
# openclaw-sandbox.yaml 예시
egress:
  allowed:
    - host: "api.github.com"
      port: 443
    - host: "api.slack.com"
      port: 443
  denied:
    - host: "*"
      port: "*"
```

- **정적 변경:** `openclaw-sandbox.yaml`을 편집한 후 `nemoclaw onboard`를 재실행하면 재시작 후에도 유지
- **동적 변경:** `openshell policy set <policy-file>`로 실행 중인 샌드박스에 적용하며, 세션 종료 시 초기화
- **프리셋:** PyPI, Docker Hub, Slack, Jira 등 일반적인 통합을 위한 프리셋 정책 파일 제공

> "NemoClaw의 보안 철학은 기본 차단, 명시적 허용(deny by default, allow by policy)입니다. 새 직원에게 처음에는 아무 권한도 안 주고, 업무에 필요한 것만 하나씩 열어주는 것과 같습니다."
